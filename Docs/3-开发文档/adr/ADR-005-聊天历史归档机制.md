# ADR-005 · 聊天历史归档机制（chatMessages 加 archived 字段 + Dexie v1→v2 升级）

- **状态**：现行（zlbdh 2026-05-09 批 PROP-002 后实施）
- **日期**：2026-05-09
- **决策人**：zlbdh / 咪咪
- **关联**：F-203 / [PROP-002](../../../确认改动/已审批/已完成/PROP-002-2026-05-09-聊天历史归档.md)

## 背景

`chatMessages` 表无任何过期 / 归档机制：所有用户与 AI 的对话永久保留，启动时 `loadSnapshot` 一次性灌进 `data.chatMessages`，由 `HomeChat` 取 `slice(-4)` 渲染。

随着使用，记录会涨到几千条：
- 启动时一次性反序列化压力
- 首页只想看到最近 4 条「活跃」对话，不想被旧消息夹杂
- 但也希望保留全部历史 — 用户能"翻历史"，备份能找回

候选方案：
- A. 硬删除超过 30 条的旧消息 → 数据丢失，与"备份找回"矛盾，**否决**
- B. 软标记 `archived` 字段 + 首页只看 active 30 内的最近 4 条 + 用户可翻全部 → **采纳**
- C. 拆出独立 archivedChatMessages 表 → 改动面更大、读写两套 schema，性价比低，**否决**

zlbdh 在 B 上再追加：**让用户能"翻历史聊天"，不只能在备份里找** — 升级原 PRD 范围，多一个「过往的话」Card 入口 + ChatHistoryDrawer 全屏抽屉。

## 决定

**给 chatMessages 表加索引化的 `archived` 字段（0|1），通过 Dexie v1→v2 schema 升级 + upgrade() 迁移把存量数据 backfill 为 archived=0。**

具体：

1. **Dexie 升级 v1 → v2**
   ```js
   db.version(2).stores({
     ...,
     chatMessages: 'id,createdAt,archived',  // 唯一变化：加 archived 二级索引
     ...
   }).upgrade(async tx => {
     await tx.table('chatMessages').toCollection().modify(msg => {
       if (msg.archived === undefined) msg.archived = 0;
     });
   });
   ```
   其他 11 张表的 schema 字符串保持原样（v2 必须列全所有表，否则丢）。

2. **字段语义**
   - `archived: 0` = 活跃 / 在首页可见 / 默认值
   - `archived: 1` = 已归档 / 首页不显示 / 仍可在 Profile「过往的话」抽屉里翻
   - 用 0|1 整数而不是 boolean，因 IndexedDB 不索引 bool

3. **归档触发：每次新消息入库后立刻**
   - useAppData 的 `addChatMessage(input)` 动作内：
     - addRecord 完成后查 `db.chatMessages.toArray()`
     - 数 `archived !== 1` 的条数（即 active 数）
     - 若 active > 30，将最老的 `(active - 30)` 条标 archived = 1（一次到位，不留尾巴）
   - 不在 loadSnapshot 里做归档（避免静默批量改用户数据）

4. **数据流**
   - `loadSnapshot`：`db.chatMessages.toArray()` 全量返回（含归档），由组件 filter
   - 首页 `recentChat`：`data.chatMessages.filter(m => m.archived !== 1).slice(-4)`
   - Profile「过往的话」Card：从 `data.chatMessages` 直接计 active / archived 数
   - ChatHistoryDrawer：从 `data.chatMessages` 全量按 createdAt desc 排序，归档项加视觉标记
   - 备份 `exportAllData`：`db.chatMessages.toArray()` 不变（含归档全量），PRD 验收第 2 条天然满足

5. **新增 UI**
   - `{{APP_REPO_DIR}}/src/features/profile/ChatHistoryDrawer.jsx`：全屏 slideup 抽屉
   - Profile.jsx 内联加一张 Card：标题「过往的话」/ 副文「N 条 · X 条已收进过往」/ 按钮「翻翻看」
   - 视觉规范：归档消息加「· 过往」标记，opacity 0.65，text-xs（11px），color inkSoft（`#3a3128`）

## 后果

### 好处
- **数据完整**：所有历史聊天保留，备份导出 JSON 含归档全量
- **首页性能**：首页 render 复杂度从 `O(N)` 变 `O(min(N,30))`，N 大时显著
- **用户能力**：新增"翻过往"入口，比 PRD 原始范围更友好（复用了 Drawer 视觉模式）
- **schema 严谨**：archived 进二级索引，未来若需高频 `where('archived').equals(0)` 查询直接可用
- **存量数据安全**：upgrade() 把所有老消息显式标 archived=0，查询语义一致

### 代价
- **Dexie version 从 v1 跳 v2**：以后再改 schema 必须 v3，不能回头
- **新增 1 个 React 组件 + 1 个 Card 入口**：Profile.jsx 增加 ~30 行（仍 < 5KB）
- **每次发消息多 1 次 db 读** + 可能 1 次 db 写：IndexedDB 上 < 5ms，可忽略
- **归档计数显示有刷新延迟**：抽屉打开后期间用户操作其他 tab 不会实时同步 count（罕见、可接受）

### 后续如果反悔了
- **撤销路径**：
  1. 删 ChatHistoryDrawer + Profile 的 Card → 用户失去入口（数据仍在）
  2. 删 Home.jsx 的 archived 过滤 → 首页又能看到全部历史
  3. 不删 archived 字段（保留向前兼容）
- **预估翻案成本**：1h 内（删 ~80 行代码 + 1 个组件 + 1 个 Card 入口）
- **数据无损**：所有 archived=1 的记录原样保留，撤销后变回普通 active

## 备注

- 归档操作放在 add 之后而不是 add 前 — add 自身的失败不影响归档逻辑
- `archived` 字段值用 `0`/`1`（数字）而不是 `false`/`true`（布尔），因 IndexedDB 不索引布尔
- 如未来要"按周/按月归档"或"7 天没动的自动归档"，可在 addChatMessage 旁加并行规则
