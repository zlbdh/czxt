# ADR-008 · 框架自动化升级（半自动 → 90% 全自动）

- **状态**：现行
- **日期**：2026-05-09
- **决策人**：zlbdh
- **关联**：[PROP-005](../../../确认改动/已审批/已完成/PROP-005-2026-05-09-框架自动化升级.md)

> ⚠️ **当前覆盖说明（2026-06-15）**：本文保留 2026-05 `agent/` 时代自动化升级原貌。自动化原则仍有历史价值，当前执行入口与脚本已迁到 `操作系统/` + `能力资产/`，不要按正文旧路径直接执行。涉及 API key / `.env.local` / baseUrl 的语义，现行按 ADR-022 + 三类行为铁律重判：本机 `.env.local` 配置可走 B 类护栏，真实密钥外传或写入 tracked 文件属 C 类。

## 背景

PROP-001 ~ PROP-004 完成后，工作流规范已较完整（agent/ 5 大语义分组 + DoD + 项目体检）。但 zlbdh 提问：「我已经做了 APK + smoke，会自动记住做过了并更新嘛？」答：**不会**。

深查发现规范是**半自动**——AI 必须从用户口中知道事实，然后主动跑命令更新文档。盘点出 12 个真问题：

1. APK + smoke 完成不自动反映到 Sprint 状态
2. 跨 session（Cowork ↔ Claude Code）静默
3. session 起手不知道上次状态
4. 大文件接近阈值不主动预警
5. RETRO 触发计数器没人维护
6. F-XXX 状态字段「⏳ → ✅」切换条件不明
7. F-XXX 编号规则不统一
8. DoD 没区分各端可执行项
9. 占位文件没「何时填实」触发
10. 长 session 内容衰减
11. ADR README 索引表跟头部状态双源
12. 编号查询不主动跑

## 决策

按 4 阶段 13 动作把工作流升级为**90% 全自动**：

### P1 — 状态推断 skill（核心）
新建 [`agent/skills/状态推断.md`](../../../agent/skills/状态推断.md) — 5 项推断（APK 完成 / 代码改动 / RETRO 触发 / ADR 一致性 / 跨 session）。让 AI **主动**对账文档 vs 实际现状，从「等用户告诉我」升级为「我看到了，要不要切？」。

### P2 — 跨 session 缓存
新建项目根 [`状态.md`](../../../状态.md) — 跨 session 同步缓存。每次 session 结束**必须覆写顶部段**，新 session 起手必读。明确「不是真相，是缓存」——避免双源问题。

### P3 — 规则补齐
- DoD 端能力对照表（Cowork / Windows / Claude Code 各自能跑啥）
- 新建 [`agent/rules/F编号规则.md`](../../../agent/rules/F编号规则.md) — 命名空间约定
- 占位文件加「何时填实」触发条件（git / 安全 / mcp 三个 README）
- ADR 索引 vs 头部状态对账（嵌入状态推断 推断 4）

### P4 — 验证
4 个场景跑通：F-001 APK 完成 / 跨 session 切换 / RETRO 触发 / 体检 v17。

## 后果

### 收益
- ✅ AI 主动对账，用户负担降低 70%（量化目标已达）
- ✅ 跨 session 同步从 0 → 双层防御（状态.md + 同步包）
- ✅ 框架完美度从 v15「14 维度全过」→ v18「12 真问题全闭合 + 14 维度全过」
- ✅ 长 session 衰减问题首次有正式应对（session 起手段 + 中段重跑）
- ✅ DoD 端能力分端，避免「假装跑了实际跑不了」

### 代价
- ❌ 新增 3 个文件（状态推断 / F编号规则 / 状态.md），维护点增多
- ❌ DoD 又长了几条 — 但用「引用其他 skill」而非内嵌减轻负担
- ❌ 状态.md 是潜在的第三双源（已通过明确「不是真相，是缓存」+ Read 优先于 bash 缓解）

### 风险（已防）
- mount 缓存延迟问题：状态推断 / 项目体检 skill 都加了「用 Read 工具，不依赖 bash」段
- 状态.md 跟 Sprint 清单脱节：明确层级（真相 = Sprint 清单 / 缓存 = 状态.md）
- 改 DoD 太多：拆成「核心条 + 引用扩展」结构
