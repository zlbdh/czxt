## 背景

ADR-009（PROP-006）完成后，v22 体检 zlbdh 追问「框架还有问题隐患吗」。盘点出 14 个潜在隐患：
- 4 个真隐患（短期可能发生）
- 5 个监控级（RETRO-003 已列）
- 3 个长期演进（不紧急）
- 2 个元层级（不可彻底解决）

zlbdh 决定「解决所有」——本 ADR 封堵 **9 个可解决的**（4 真 + 5 监控）。**5 个不可解决的接受现状**。

## 决策

### P1 — 真隐患 4 项

#### P1-1 状态.md 多 session 写冲突
**问题**：Cowork 和 Claude Code 同时改 状态.md → 后写覆盖先写。
**解决**：状态.md 加「写入约定」段——写之前必 Read 看 mtime，跟「上次读」不一致 → merge 而非覆写。zlbdh 是冲突仲裁人。

#### P1-2 F-001 用户主动报告依赖
**问题**：用户做完 APK + smoke 不主动告诉 AI，状态字段过期。
**解决**：状态推断 推断 1 强化——AI **每次响应必跑**（不只是 session 起手），看到 ⏳ 待 APK + 最新 APK > 代码完成时间 → 主动问。dismiss 后 7 天 cooldown。

#### P1-3 8 大文件接近 7KB 无主动拆分
**问题**：PROP-001 拆完后文件爬回 6000-6741B，8 个接近 7KB，没主动拆分触发。
**解决**：
- 项目体检 维度 1 加「6500B 必拆门槛」分级
- agent/rules/写代码.md 加硬规则「≥6500B 必须立刻开 PROP 拆分，不等扩展」

防御「拆完→涨回→再拆」无限循环。

#### P1-4 AI 误删无恢复机制
**问题**：sandbox 不直连 git，误删后无标准流程。
**解决**：agent/rules/安全与隐私.md 加「误删恢复流程」段——立刻停手 / 告知 zlbdh / 不重写凭记忆 / git restore 优先。

### P2 — 监控级 5 项（RETRO-003 列出）

#### P2-1 推断 3 阈值算法精修
**问题**：原算法 `3 × (retro_count + 1)` 在 RETRO 提前/延后写时失准。
**解决**：改为**增量计数** — `total_l3 - 上次 RETRO 涵盖数 ≥ 3`。

#### P2-2 RETRO 触发节奏 3 ± 1 灵活
**问题**：「严格 3 个」太死板。
**解决**：3 ± 1 窗口。2 个可提前（密集架构改动），4 个可延后（简单改动）。决策权在 zlbdh。

#### P2-3 PROP 卡进行中 > 7 天告警
**问题**：进行中 PROP 没限期，可能被遗忘。
**解决**：状态推断 加 推断 6——mtime > 7 天 告警「卡了 N 天，要不要弃用 / 拆分」。

#### P2-4 占位文件主动监控
**问题**：git/安全/mcp 占位文件的「何时填实」要主动判断，但没机制。
**解决**：状态推断 加 推断 7——实施时扫描改动是否触碰相关关键词，主动提示。

#### P2-5 状态.md 建议决议表清理机制
**问题**：表会越来越长，无清理。
**解决**：7 天前自动归档到「📦 历史归档」段。dismissed 项 7 天后冷却结束可重推。

## 接受不解决的 5 个

- #14 认知负担过重（设计选择，不是 bug）

## 后果

### 收益
- ✅ 4 个真隐患全封堵 — 短期内不会发生

## 立场声明

ADR-010 **不假装彻底完美**。承认：
- **5 个不可解决** —— 多人协作 / 迁移 / 备份 / 自我引用 / 认知负担。这些不是 bug 是设计取舍。
- **3% 不自动** —— AI 是响应式不是 daemon，永远不可能 100%。
- **大文件门槛 6500B 不立刻补拆现有 8 个文件** —— 等扩展触发。

承认局限是工程成熟，不是缺陷。
