## 背景

ADR-010 P1-3 设立硬规则：**源文件 ≥6500B 必拆，不等下一次扩展**。但 ADR-010 同时声明「不立刻补拆现有 8 个文件，等扩展触发」。

PROP-006/007 实施过程中，6 个文件已经爬过 6500B 门槛：

| 文件 | 字节 | 越线原因 |
|---|---|---|
| `agent/skills/状态推断.md` | 9715B | PROP-005 新建后 PROP-006/007 反复扩展（推断 5-7 / 算法精修）|
| `agent/workflows/实施循环.md` | 9309B | DoD 核心 5 项 + 跨 session 同步包 + 端能力对照表反复加 |
| `agent/skills/项目体检.md` | 8078B | PROP-005 新建后 PROP-006/007 加 6500B 阈值表 + 跨 session 维度 |
| `{{APP_REPO_DIR}}/src/shared/DayDrawer.jsx` | 6741B | PROP-002 聊天历史归档时加 ChatMessageRow + 大量分组渲染逻辑 |
| `agent/rules/改动分级.md` | 6694B | PROP-005/006 加 PROP 状态字段写法 + L1/L2 边界细则 |
| `{{APP_REPO_DIR}}/src/features/timeline/Timeline.jsx` | 6564B | PROP-003 导航重构时加月历占位 + 录入交互 + summary 渲染合体 |

F-001 收档体检发现后，硬门槛已经被全员越线 — 必须实施拆分。

## 决策

按「主文件保留核心 + 拆出辅助文件」模式拆分。**保持功能、路由、数据结构不变**。

### 文档类（4 个）

| 主文件保留 | 拆出文件 | 拆分维度 |
|---|---|---|
| `状态推断.md`（入口 / 7 项索引表）| `状态推断-推断项.md` + `状态推断-跨session监控.md` | 推断 1-4 走推断项 / 推断 5-7 走跨 session 监控 |
| `实施循环.md`（流程主干）| `实施循环-DoD.md` | DoD 完整清单（L1-L4 + 跨 session 同步包）|
| `项目体检.md`（入口 / 跑法 / 报告模板）| `项目体检-检查项.md` | 5 项检查的完整逻辑/命令/判定模板 |
| `改动分级.md`（L1-L4 判定）| `改动分级-状态字段.md` | PROP 状态字段写法 + 状态机 |

### 代码类（2 个）

| 主文件保留 | 拆出文件 | 拆分维度 |
|---|---|---|
| `DayDrawer.jsx`（弹层壳 / 关闭 / 组合）| `DayDrawerBody.jsx` + `DayDrawerRows.jsx` | Body 负责分组渲染（睡眠/进食等行）/ Rows 负责文本/聊天消息单行渲染 |

**注**：PROP-008 提案中曾写 `DayDrawerGroups.jsx`，实施时改名 `DayDrawerBody.jsx`（更贴合实际职责：渲染整个 body 而非单纯分组）。

## 后果

### 收益

- ✅ 6 个 PROP-008 范围文件全部回到 < 6500B 安全区
- ✅ 主文件职责更清晰（DayDrawer = 壳，Timeline = 状态容器）
- ✅ 拆出的辅助文件单独维护，未来扩展不会跨边界
- ✅ ADR-010 P1-3 硬规则首次落实到「主动拆」而非「警戒不动」

### 代价

- ❌ agent 文档从 4 个变 8 个（认知负担）— 但 INDEX 表压力降低，定位反而更快
- ❌ Timeline 拆出 3 个子组件后 import 链稍长，但路由/数据流不变
- ❌ 历史 PROP/ADR 内文引用旧路径（如 `状态推断.md 推断 5`）按 ADR-007 §C 不动，靠 redirect 注释帮助查找

### 风险（已防 + 新发现）

- ✅ 拆出的代码组件用 `wc -lc` + esbuild + vitest + vite build 四层验证
- ✅ 拆出的 md 文件主文件加「关联」段反向链接
- 🔴 **DoD 文件 PROP-009 后越线** — `实施循环-DoD.md` 拆出时是 ~5KB，PROP-009 加交接卡 6 段后涨到 7029B。**这是 PROP-008 拆完后新增的越线**，不在 PROP-008 范围内，建议新开 PROP-010 处理（把交接卡段独立成 `交接卡格式.md`）。
- 🟡 **`项目体检-检查项.md` 6200B 在警戒区** — 接近门槛，下次扩展前先 `wc -c`，必要时拆出「6500B 阈值历史」段

## 立场声明

ADR-011 是 **ADR-010 P1-3 的落地验证**。

- ADR-010 设规则但暂缓补拆（「等扩展触发」）— 留下灰色地带
- ADR-011 把 6 个历史越线文件一次性拆完 — 灰色地带清除
- 拆完不是终点：拆出的辅助文件本身也可能再涨过门槛（DoD 7029B 已经证明），需要持续监控（项目体检 检查项 1）

**6500B 门槛不是「拆一次就没事」的一次性工作，是持续的卫生规则。**

## 一句话

ADR-011 把 ADR-010 P1-3 的硬规则从「警戒不动」推到「全部拆完」，并暴露出新的循环增长风险（DoD 7029B）— 6500B 门槛是持续运营而非一次清理。
