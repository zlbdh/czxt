# ADR-017 · 6500B 文件大小规则按工具分层化（PROP-013）

- **状态**：现行
- **日期**：2026-05-12
- **决策人**：zlbdh
- **关联**：[PROP-013](../../../确认改动/已审批/已完成/PROP-013-2026-05-12-6500B规则按工具分层化.md) · 修订 [ADR-010](ADR-010-框架自动化v3-9隐患全封堵.md) P1-3 段 · 不替代 [ADR-011](ADR-011-6500B文件拆分.md)（历史拆分本身合理保留）

## 背景

PROP-007 / ADR-010 P1-3 立了「单文件 ≥ 6500B 必拆门槛」硬规则，PROP-008 / ADR-011 一次性把 6 个越线文件拆完。规则跑了一段时间后 RETRO-005 暴露**规则跟工具能力脱节**：

### 触发证据（来自 RETRO-005）

| # | 信号 | 严重 |
|---|---|---|
| 1 | bash `wc -c` mount 缓存反复制造"幻警戒"（Health.jsx 真实 6110B 被读成 6273B 多个 session）| 🔴 卡 2 |
| 2 | Cowork-specific 限制泛化到 Claude Code / Codex — 他们写文件没 mount 限制，被人为限制 | 🔴 卡 3 |
| 3 | 多次"为了迁就 Cowork"做无意义抽组件 — 违反 YAGNI | 🟡 |
| 4 | PM 自己也会被规则误导（F-LAYOUT-2 差点让 Claude Code 抽 HealthExercises）| 🟡 |
| 5 | 跟 ADR-016「按工具分层」模式不一致 | 🟡 |

### 元层级洞察（RETRO-005 洞察 1）

PROP-011 / PROP-012 / PROP-013 同一治理模式：
**「单工具/单维度限制被错误泛化到多工具协作 → 噪音 → 拆到合理粒度」**

## 决策

把 6500B 规则从「所有 AI 都遵守必拆门槛」改成**按工具分层**：

### 阈值分级

**Cowork（mount 截断风险，9KB 硬截断）**：

| 字节区间 | 状态 | 动作 |
|---|---|---|
| < 6000B | ✅ 安全 | Edit / Write 都可用 |
| 6000-6500B | 🟡 警戒 | Edit 前 Read 复核；改大文件建议 Python/Bash 写入 |
| 6500-8000B | 🟡 软建议 | **必须用 Python/Bash 绕 Edit**；不强制开 PROP 拆 |
| ≥ 8000B | 🔴 危险（接近 9KB 硬截断）| Cowork 不要用 Edit；交 Claude Code / Codex 改 |

**Claude Code / Codex（本地文件系统，无 mount）**：

| 字节区间 | 状态 | 动作 |
|---|---|---|
| < 8KB | ✅ 安全 | 无限制 |
| ≥ 8KB | 🟡 软建议抽组件 | 工程审美层面建议拆（单一职责），**不阻塞实施** |

### bash `wc -c` 关键决策强制 Read 复核

RETRO-005 卡 2 修复 — bash 通过 mount 读取**可能返回陈旧缓存**。

→ 关键决策点（决定是否抽组件 / 触发警戒动作）必须用 **Read 工具复核**真实字节值：
- bash + Read 不一致时 → 信 Read（bash 是 mount 缓存）
- 已写入 `agent/skills/项目体检-检查项.md` 维度 1 命令模板

### 保留的工程审美约束

所有工具单文件 **>8KB 软建议抽组件**（单一职责），这是真实工程质量约束，跟工具能力无关。

## 后果

### 收益
- ✅ 消除"幻警戒"反复出现（bash mount 缓存 → Read 复核兜底）
- ✅ Claude Code / Codex 写文件不被人为限制（< 8KB 软建议）
- ✅ Cowork 仍有 mount 安全裕度提示（6500B 软建议 + Python/Bash 绕行方案）
- ✅ 跟 ADR-016 模式一致（按工具分层治理）
- ✅ 保留 8KB 工程审美约束（单一职责，跟工具无关）

### 代价
- ❌ 阈值分层增加认知负担（2 套阈值表）
- ❌ 项目体检告警变弱 → 双源复核 + 工具上下文缓解

### 风险（已防）
- ✅ Claude Code / Codex 写出 7-8KB 巨型文件 → 8KB 软建议抽组件兜底
- ✅ Cowork 不小心用 Edit 改 >6500B 文件触发截断 → 阈值表 + 警戒提示 + 用 Python/Bash 替代方案
- ✅ bash 缓存陷阱再次出现 → 强制 Read 复核已写入命令模板

### 不动的部分
- ✅ ADR-010 P1-3 历史背景**保留**（解释当年为什么立 6500B 门槛）
- ✅ ADR-011 / PROP-008 历史拆分**保留**（单一职责本身合理）
- ✅ Cowork 改 >6500B 文件**仍要绕 Edit**（用 Python/Bash 写入是真实物理约束的对策）
