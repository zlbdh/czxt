# ADR-018 · 流程优先 + 交接卡强制（PROP-014）

- **状态**：现行
- **日期**：2026-05-12
- **决策人**：zlbdh
- **关联**：[PROP-014](../../../确认改动/已审批/已完成/PROP-014-2026-05-12-流程优先与交接卡强制.md) · 补 [ADR-012](ADR-012-交接卡机制.md) + [ADR-013](ADR-013-交接卡格式独立拆出.md) + [ADR-015](ADR-015-交接区机制.md) 三个交接卡相关 ADR 留下的缺口（chat 简版没强制 / 流程问题没规则）

> 现行执行口径：本 ADR 原始版本强制 chat 简版 ①-⑥；后续 [PROP-027 v2](../../../确认改动/已审批/已完成/PROP-027-v2-2026-05-21-chat简版⑦段硬约束.md) 已扩展为 chat 简版 ①-⑦，执行真源以 [`../../../操作系统/03_交接/交接卡格式.md`](../../../操作系统/03_交接/交接卡格式.md) 为准。

## 背景

ADR-012/013/015 完成了**交接卡内容结构**（6 段 + 拆出格式 + 交接区事件流），但留了两个缺口：

1. **流程问题没规则**：实施中遇到流程 bug（规则错位 / 警戒误报 / 路径错）时该停手修流程还是继续业务，靠 PM 主观判断
2. **chat 简版没强制**：完整交接卡文件存在 → PM/Owner 看 chat 时未必能秒识别交接卡（179 行的完整卡 vs 5 行的简版差异）

PROP-014 来补这两个缺口 + 引入 chat 简版 ①-⑥ 格式强制。

### 触发证据（来自 zlbdh 一次对话）

**信号 1**（2026-05-12 12:35）：
> 「为什么 Claude Code 没有生成交接卡？我们是不是得加个规则：
> 1、每次开发、变动等操作都要按照规范流程，如果流程有问题则先停止开发先优化流程；
> 2、如果需要交接，每次都生成交接卡片」

**信号 2**（zlbdh 追加解释）：
- 实际上 Claude Code **写了**完整交接卡到 `交接区/待接手/2026-05-12-1230-F-PROFILE-12-Claude到Codex.md`
- 但 Claude Code 在 chat 里给了"总结段"（列文件 / 测试 / 决策），**没有 ①-⑥ 简版交接卡格式**
- → 视觉信号缺失导致 PM 误以为「漏交接卡」

**信号 3**（PM 自反）：
- 之前 Codex 完成 F-LAYOUT-12 / F-003 / F-002 等都给了完整 ①-⑥ 简版（zlbdh 看截图能秒懂）
- Claude Code 这次 F-PROFILE-12 改用了"总结"格式 — **工具间行为不一致 = 没规则兜底**

## 决策

引入 **3 条元规则**：

### 规则 1：流程优先三阶判定

写入 `agent/workflows/实施循环.md`：

| 严重度 | 表现 | 行为 |
|---|---|---|
| 🔴 阻塞 | 影响代码 / 测试 / 实施落地 | 立刻停手修流程 → 开 PROP 治理 → 修完再继续 |
| 🟡 非阻塞 bug | 警戒误报 / 文档不一致 / 路径错 | 业务继续 + 立刻进 RETRO backlog |
| 🟢 优化机会 | "换种方式可能更好" | 业务继续 + 下次 RETRO 讨论 |

→ 平衡「不让流程绑架业务」vs「不让业务掩埋流程债」。

### 规则 2：交接卡强制矩阵

写入 `agent/workflows/实施循环.md` + `agent/workflows/交接卡格式.md`：

| 行为类型 | 必写完整交接卡？ | 必发 chat 简版 ①-⑥？ |
|---|---|---|
| 跨工具切换（Cowork ↔ Claude Code ↔ Codex ↔ zlbdh）| 强制 | 强制 |
| 实施完成（写代码 / 改 framework / QA / push）| 强制 | 强制 |
| PM 内部推进（写 PRD / 选方案 / 答问题）| 只更新 状态.md | 可选 |
| 临时讨论（confirm / 简单问答）| 不需要 | 不需要 |

### 规则 3：chat 简版交接卡 ①-⑥ 格式强制 ⭐

写入 `agent/workflows/交接卡格式.md`。**实施完成时除写完整交接卡文件，还必须在 chat 里发一段简版 ①-⑥**。

标准格式（5-10 行紧凑版）：
```
【交接卡 - <任务名> 从 <A> 到 <B>】
① 时间：YYYY-MM-DD HH:MM
② 文件变更：N 新建 / M 修改（含关键 commit hash / 字节体检 / APK 路径）
③ 测试：vitest ✅/❌ N/N | esbuild ✅/❌ | build ✅/❌ | APK ✅/⏳ | smoke ✅/⏳ | push ✅/⏳
④ 你要做：[1] xxx [2] xxx [3] xxx
⑤ 警戒：✅/🟡/🔴 关键警戒
⑥ 详情：读 交接区/待接手/...md
```

**❌ 不接受"总结段"代替**（即使内容详细）— PM 看 chat 必须看到 ①-⑥ 编号，跟完整卡 6 段对齐。

## 后果

### 收益
- ✅ 流程问题严重度有规则判定 — 不靠 PM 主观决定
- ✅ chat 简版强制 — 工具间行为一致，PM 视觉信号不缺失
- ✅ "总结段不能代替"硬规则 — 防止 Claude Code 类工具偷懒
- ✅ 跟 PROP-011/12/13 模式呼应：framework 演进**第 4 次按模式治理**（这次是"补缺失规则"不是"修错位规则"）

### 代价
- ❌ 实施完成时多写一段 chat 简版（5-10 行，约 30 秒开销）— 但本来就该有
- ❌ 流程优先判定增加认知负担 — 但分三阶比一刀切「有问题就停」更合理

### 风险（已防）
- ✅ Claude Code / Codex 忘记发 chat 简版 → PM 起手 5 步会看交接卡，发现没简版可以追问
- ✅ 流程优先判定标准不一致 → 三阶判定表给了具体例子（PROP-014 实例：路径自纠 = 🟡 / ProfileCard 越软建议 = 🟢）
- ✅ 简版格式不统一 → 标准模板写在 交接卡格式.md，新文件 / Codex 自动遵守

### 不动的部分
- ✅ 完整交接卡 6 段格式（PROP-009/010/011 已稳定）保留
- ✅ 交接区目录结构（待接手/已接手/历史归档）保留
- ✅ 状态.md 顶部摘要段保留
- ✅ AI 边界 A/B/C 三类不动（那是行为类别，不是流程规范）
