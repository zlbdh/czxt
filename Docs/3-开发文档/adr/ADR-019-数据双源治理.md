# ADR-019 · 数据双源治理：profiles 合并到 userProfile（PROP-015）

- **状态**：现行
- **日期**：2026-05-12
- **决策人**：zlbdh
- **关联**：[PROP-015](../../../确认改动/已审批/已完成/PROP-015-2026-05-12-数据双源治理.md) · Sprint-3 业务前置阻塞解除

## 背景

Sprint-2 引入 `userProfile` 作为「我」页 5 段档案的数据源，但早期 schema 仍保留 `profiles`：

- `profiles`：v1-v2 旧身体档案，字段含 `height` / `weight` / `bodyFat` / `goal`
- `userProfile`：v3 新档案，字段含身体、工作节奏、饮品习惯、目标、咪咪人格模式

两张表并存带来 3 个问题：

1. UI 上可能同时出现旧「身体档案」和新「个人档案」
2. 字段重叠时不知道该读哪张表，尤其 `height` / `weight`
3. Chat / Plan 等 Sprint-3 业务只读 `userProfile`，旧 `profiles` 里填过的数据会失效

## 决定

从 v3.2 起，应用读写层统一以 `userProfile` 为唯一档案源。

### 1. Dexie schema 升到 v4

新增 v4 migration：启动时读取旧 `profiles.get('main')`，转换后合并进 `userProfile.get(1)`。

v4 暂时仍 declare `profiles` store，不物理删除旧表。原因是移动端 IndexedDB 迁移失败时需要保留旧数据，等 v3.3/v5 验证稳定后再考虑 drop store。

### 2. 字段映射

| 旧 profiles | 新 userProfile | 策略 |
|---|---|---|
| `height` | `height` | 原样搬 |
| `weight` | `weight` | 原样搬 |
| `bodyFat` | `bodyFat` | 新增到 ProfileCard 基础身体段 |
| `goal: "recomp"` | `mainGoal: "目标A"` | enum 映射 |
| `goal: "cut"` | `mainGoal: "减脂"` | enum 映射 |
| `goal: "bulk"` | `mainGoal: "增肌"` | enum 映射 |
| `activity` | 不迁移 | 暂不进入 userProfile，`calcTargets` fallback 到 `moderate` |

### 3. mergeFillEmpty 策略

迁移时使用 `mergeFillEmpty(existing, patch)`：

- 只填 `existing` 为空的字段
- 不覆盖用户已经在新 ProfileCard 填过的字段
- 迁移函数失败时 catch 并保留旧 `profiles` 数据

这保证 v3.0/v3.1 已经填过 `userProfile` 的用户不会被旧数据回退。

### 4. 双向 enum 兼容期

`calcTargets` 与 `analyzeMealImage` 在 v3.2 保留短期双向兼容：

- 旧字段：`goal = cut | recomp | bulk`
- 新字段：`mainGoal = 减脂 | 目标A | 增肌 | 维持 | 其他`

所有调用方已改为传 `data.userProfile`，但兼容分支保留一个版本，降低升级和备份导入风险。后续可在 RETRO-007 或 v3.3 评估删除旧 enum 分支。

## 后果

### 收益

- 「我」页只剩一张 ProfileCard，用户不再面对两个档案入口
- Chat / Health / Home / AI 餐照分析统一读 `userProfile`
- 旧备份或旧 `profiles` 数据可自动迁入新档案
- `database.js` 迁移逻辑拆到 `databaseMigration.js`，避免 database 文件继续膨胀

### 代价

- v4 仍保留 `profiles` store，底层物理债务未完全清掉
- `calcTargets` 需要短期保留旧 enum 分支
- `bodyFat` 加入基础身体段后，旧用户完整度可能从 5/5 临时变成 5/6，需要补填体脂率

### 风险与缓解

- 迁移失败：v4 upgrade 包 try/catch，旧 `profiles` 不删，下次仍可重试
- 新旧字段冲突：`mergeFillEmpty` 只填空，不覆盖新档案
- 备份导入旧 profiles：`importAllData` 后调用 `tryMigrateImportedProfile`
- 真机验证：v3.2 APK 已安装，Dexie v4 打开无报错，`userProfile.mainGoal` 被 Chat 正常读取

## 一句话

ADR-019 把档案数据源从「`profiles` + `userProfile` 双源并存」收敛到「`userProfile` 单源读写」，但物理层保留旧 `profiles` 一个版本作为迁移保险。Sprint-3 可以在这个单源基础上继续做计划、日程和更细的 AI 上下文。
