# ADR-038 · PM 实体化 agent 调度模型（PROP-044 升永久）

- **状态**：现行
- **日期**：2026-06-14
- **决策人**：zlbdh approve（4 次追问"为什么不开 agent"驱动）/ 沉淀 PM 起草
- **关联**：[PROP-044](../../../确认改动/已审批/已完成/PROP-044-2026-06-14-framework写入实体化与PM-agent化评估.md)（升永久）· [RETRO-021](../../7-复盘/RETRO-021-2026-06.md)（B-lite 首次试点）· [RETRO-022](../../7-复盘/RETRO-022-2026-06.md)（本批复盘）· [ADR-027](ADR-027-议题CU+DD永久化-沉淀PM元层架构.md)（9 PM 四层）· [ADR-031](ADR-031-项目PM对外身份完整铁律.md)（项目 PM 单点对外）· [`子agent调度机制.md`](../../../操作系统/01_架构/子agent调度机制.md) §三.5 + §四

> 术语现行统一：早期“主-元-子-子子”已统一为“主-元-决策-实施”；本 ADR 保持 PM agent 调度模型不变。

## 背景

ADR-027 定了 9 PM × 主-元-决策-实施四层的**角色模型**，但 PM「做活」长期靠主会话亲自写——角色是抽象的，没实例化成真实 agent。本日 zlbdh 4 次追问"为什么不开 agent"，根本问题是：framework 写入默认单点（主会话手改），既慢、又没把"写"和"验收"分开。需要把角色模型**落地为 agent 调度模型**：每个 PM 的工作默认实例化为真实 agent，同时不能松动 ADR-031「项目 PM 唯一对外 + 单点拍板」和 ADR-027 的层级铁律。

关键约束（决定性）：**worktree 隔离对 framework 不适用**——项目根（{{PROJECT_ROOT}}）非 git 仓库，多 worker 改 framework 无法靠 git worktree 物理隔离，只能靠"写集互斥声明 + 主会话单点合并 + 体检 gate"做受控并行。

## 决定

把 PROP-044 升为永久 ADR，固化 **PM 实体化 agent 调度模型**（第 23 元规则 **DX**）：

### 1 — 9 PM 全量 agent 化

每个 PM 的工作**默认实例化为真实 agent**：**写=worker / 只读=explorer**。
- 项目 PM「咪咪」**不实例化**——它本身是调度者 / 主会话。
- 其余 8 PM 切角色做活时按上述映射开 agent（见 `子agent调度机制.md` §四 PM→agent 映射表）。

### 2 — 项目 PM 单点调度 + 验收（ADR-031/027 一脉）

- **spawn 权 + 验收权恒在项目 PM「咪咪」**，各 PM 不自治乱开、不下放 spawn 权。
- 对外身份永远项目 PM「咪咪」（ADR-031 不变）；其他 agent 是内部信号、不独立对外署名。
- **验收链恒平、单点在项目 PM**——无中间未验收层。

### 3 — 单源禁并行文件：agent 起草 / 主会话落最后一笔

`状态.md` / `CHANGELOG` / **`元规则池.md`** / `角色边界.md` 是单源单点禁并行文件：worker 只**起草**，主会话**落最后一笔**，永远不派并行 worker 直接写。

### 4 — 禁嵌套自治派 agent / 验收恒单点

worker **不自己再 spawn 子 agent**。worker 需要帮手 → 回流到项目 PM，由项目 PM 派**平级** agent 并亲自验收。
- 禁嵌套的理由：嵌套会让"父 worker 的一线验收"无人监督、稀释验收质量（护 ADR-031 单点）。
- B-lite「项目 PM 派多个**平级** worker」**≠ 嵌套**（嵌套 = agent 再派 agent）——前者允许、后者禁止。

### 5 — B-lite 受控并行（子机制）

framework 写入默认主会话单点；**仅当 ≥6 个互斥文件大批量**时才派多 worker 并行：
- 写集按子目录/文件**互斥**切分（写集互斥声明派工卡必填）；
- 主会话**单点合并** + 体检 **gate**（`check-operating-system.ps1` exit 0 才接纳）；
- 单源禁并行清单永远单点（见决定 3）。
- 见 `子agent调度机制.md` §三.5。

### 6 — 元规则池升级 22 → 23（第 23 元规则 DX）

## 后果

### 好处
- PM「做活」从主会话亲手写 → 默认派 agent，写/验收分离，吞吐与质量双升。
- 角色模型（ADR-027）落地为可执行的调度模型，"为什么不开 agent"有了固定答案。
- 单源落笔 + 禁嵌套 + 验收恒单点，把 ADR-031 单点铁律延伸到 agent 时代，不被并行稀释。

### 代价
- 主会话承担全部 spawn + 合并 + 验收负载（单点调度的代价）；大批量受控并行有 barrier 浪费风险（见候选 DU）。
- 受控并行多一轮独立验证（scan + 抽检 + 体检），小批量（<6 文件）可能吃掉并行收益 → §三.5「≥6 文件」阈值兜底。
- worktree 隔离不可用 → 互斥靠"声明 + 验收"而非物理隔离，依赖派工纪律（DV 越界探针缓解）。

### 后续如果反悔了
- 退回"主会话单点亲写 framework"即可（去掉 agent 实例化），但失去写/验收分离与并行吞吐。
- 禁嵌套若放开 = 重新暴露"未监督的一线验收"风险，强烈不推荐（这是验收质量底线）。
