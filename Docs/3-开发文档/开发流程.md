## 🔄 标准循环：5 步

```
0. 接收 (Intake)
   ↓
1. 需求对齐 (PM 角色)  ← 不清楚就问，写进 PRD
   ↓
2. 方案设计 (PM + Dev)  ← UI 草图 / 数据结构 / 技术决策
   ↓
3. 实施 (Dev 角色)      ← 按文件改代码 + 写测试
   ↓
4. 验证 (QA 角色)       ← vitest + esbuild 语法 + vite build + 手动 smoke
   ↓
5. 发布                 ← APK / commit / 更新 docs/CHANGELOG.md
```

每一步都有可见产物（文档 / 代码 / 报告），不是黑箱。

## 0. 接收

zlbdh 发出新请求，咪咪先做：

- **判断类型**：
  - **小修**（< 50 行改动 / 单文件 / 视觉调整 / 文档）→ 直接进 Step 3
  - **新功能 / 重构 / 新模块**（涉及多文件 / 数据 schema / 新 UI 流）→ 走完整流程
- **如果模糊**：用 AskUserQuestion 问 1-3 个问题，定方向再走
- **建任务**：用 TaskCreate 把分解写下来，每步对应一个 task

## 1. 需求对齐（PM 角色）

输入：用户原话 + 当前代码状态 + plan 文档

输出文档：
- **PRD 条目** 加进 `docs/PRD-v3.md`（用户故事 + 验收标准 + 优先级）
- 更新 `docs/CHANGELOG.md` 的 Unreleased 段

工作步骤：
1. 用「用户故事」格式：作为 _用户_，我想要 _做什么_，以便 _达到什么目的_
2. 写 3-5 条验收标准（AC）：可勾选的具体行为
3. 标优先级 P0 / P1 / P2
4. 列出受影响的文件 / 数据表 / API
5. **如果有不确定**，必须停下来 `AskUserQuestion`，不臆测

## 2. 方案设计（PM + Dev）

输入：PRD 条目

输出：
- 数据 schema 改动（database.js / defaults.js）
- 文件清单（新增 / 修改 / 删除）
- UI 草图（用 ascii / mermaid / 文字描述）
- 风险清单：可能踩的坑

模板：
```markdown
## 设计：<feature>

### 数据
- 表名：xxx
- 字段：...
- migration：legacy → v3 怎么迁

### UI
- 入口位置：...
- 主交互流：1) ... 2) ... 3) ...
- 错误状态：...

### 代码改动
- + src/features/xxx/Xxx.jsx (new)
- M src/shared/components.jsx
- M src/shared/database.js

### 风险
- 写入文件 > 9KB → 走 outputs+cp
- ...
```

## 3. 实施（Dev 角色）

按文件改。**记 mount 写入限制 9105B**：

- < 8KB：直接 Write 到目标
- > 8KB：Write 到 outputs/X.jsx，bash `cp -f` 覆盖
- 大于 9KB：拆 part1+part2 写 outputs，bash `cat part1 part2 > target`
- 小改动：Python in-place `re.sub` 最稳

每次改完文件**立刻验证文件完整**：`tail -3` 看末尾、`wc -lc` 看大小。

## 4. 验证（QA 角色）

3 层验证（必须全过才算完成）：

| 层 | 命令 | 通过标准 |
|---|---|---|
| 语法 | `node syntax-check.mjs` | 全部 ✓ 0 fail |
| 单元测试 | `npm test` | 33+ tests passed |
| 生产构建 | `npx vite build` | 0 error，bundle 报告输出 |

可选第 4 层：手动 smoke（点击 5 个 tab、打卡、记一笔账、生成 AI 简报）。

## 5. 发布

- 更新 `docs/CHANGELOG.md`（移 Unreleased → vX.Y）
- 提示用户在 Windows 跑 `npm run android:build-apk`
- APK 出来 → smoke test → 上传到手机

## 🛠️ 已知技术约束

| 约束 | 处理 |
|---|---|
| Mount 写入限制 ~9105B | 大文件走 outputs + cp / part 拼接 |
| Edit 工具偶尔截断 | 优先 Python in-place 或 Write 完整 |
| 沙箱无 Android SDK | APK 必须在用户 Windows 上打 |
| 沙箱 npm install 慢 | 多次 35s 重试，约 60-90s 完成 |
