| 场景 | 是否适合 |

## 提示词（完整版，直接复制给目标 AI）

````
你的角色是这个项目的 PM / 架构师。我要你做一次「开发操作系统体检」 — 
不是检查业务代码，而是检查**支撑业务开发的整套规范、流程、协作机制**。

═══ 背景 ═══

你可以把项目的"开发操作系统"理解为：
- 业务代码本身（核心产品）= 🍞 面包
- 开发操作系统 = 🔪 做面包的工具、流程、规范、协作机制

很多项目花全部精力做面包，但工具没磨好 → 越做越乱、跨工具切换信息丢失、
新加 feature 时不知道按什么流程走、规则跟现实脱节。

这次体检就是给"做面包的工具"打分。

═══ 19 维度体检清单 ═══

每个维度判定 ✅ 完备 / 🟡 部分 / 🔴 缺失 / 💡 改进建议。

【A · 需求管理】
1. PRD / 需求清单是否有结构化文档？
2. 需求变更有没有版本历史可追溯？
3. 业务需求 vs 技术任务是否分离（避免 PM 直接动技术细节）？

【B · 改动分级】
4. 改动有没有等级划分（如 L1 typo / L2 bug 修复 / L3 新 feature / L4 架构）？
5. 不同等级是否对应不同审批流程？

【C · AI / 工具边界】
6. AI 工具能做什么、不能做什么有明文规则吗？
   （特别检查：git push / API key / 用户数据删除 这类敏感操作）
7. 规则是否区分「AI 自动可做」「必须先问」「永不能动」三类？
8. 敏感操作前 AI 是否被强制查规则（不靠记忆）？

【D · 跨工具 / 跨 session 协作】
9. 不同工具（IDE / CLI / 浏览器 AI / 终端）之间的交接有规范吗？
10. 跨 session 状态是否有缓存机制（避免每次重新对齐上下文）？
11. 交接信息是结构化的（如 6 段卡 + chat 简版 ①-⑥），还是散文式的"总结"？

【E · 决策档案化】
12. 重要架构决策有没有 ADR（Architecture Decision Record）？
13. 决策推翻时有没有"不修改原 ADR + 写新 ADR"的规则？

【F · 项目自检 / 健康度】
14. 有没有自动 / 半自动的项目体检机制（文件大小 / 文档一致性 / 计数对账）？
15. 有没有"状态推断"机制（AI 主动对账文档 vs 实际状态）？

【G · 复盘 / 学习】
16. 有没有定期复盘机制（如每 3 个 L3+ 一次 RETRO）？
17. 复盘提出的改进有没有跟踪机制（防止"提了不改"）？

【H · 流程跟现实的张力】
18. 当流程跟现实脱节时（规则过严 / 工具能力变化 / 用户偏好变化），有没有"反馈→识别→治理"循环？
19. 有"破例"发生时是否被记录 + 第 2 次破例时强制开 PROP 治理？

【I · 文件大小 / 工具限制】
20. 单文件大小有没有规则？规则是否按"工具实际能力分层"（避免一刀切泛化）？

═══ 输出格式 ═══

请按下面格式输出体检报告：

```
🔬 开发操作系统体检报告 — <项目名> — <日期>

【整体打分】X/19 ✅ 完备 / Y 🟡 部分 / Z 🔴 缺失

【优势】（3-5 条具体的事）
1. ...
2. ...

【关键缺失】（按优先级排序，最多 5 条）
🔴 P0 — <缺失项> — 建议立刻补 → 理由
🟡 P1 — ...
🟡 P2 — ...

【改进路径】（3 步起步方案，从最小代价开始）
Step 1（半天）: ...
Step 2（1-2 天）: ...
Step 3（1 周）: ...

【参考资料】
如想看一份"成熟开发操作系统"长啥样，可以参考另一个项目的 framework：
- agent/{agents,workflows,skills,rules,mcp}/ — 5 大语义分组
- 确认改动/{待审批,已审批/{进行中,已完成,已弃用},拒绝}/ — PROP 生命周期
- 交接区/{待接手,已接手,历史归档}/ — 跨工具交接事件流
- Docs/{1-需求文档,3-开发文档/adr,7-复盘}/ — 长期文档
- 状态.md — 跨 session 快照
- 14 个 PROP / 18 个 ADR / 5 个 RETRO 的实战档案

【一句话总结】
（这个项目的开发操作系统当前最大的杠杆点在哪里）
```

═══ 执行规则 ═══

- 不要假设项目"应该"有什么 → 看实际有什么 / 缺什么
- 不要照搬别处规范 → 给"适合本项目当前阶段"的建议
- 优先级判定标准：能否阻塞业务 / 能否避免后期翻盘代价
- 体检过程中如果发现规范文档但跟实际不一致 → 标 🟡 部分 + 说明
- 给具体建议时给"最小可行版本"，不要一上来推 14 个 PROP

═══ 你现在 ═══

请先：
1. ls / find 看下项目根目录结构
2. 读 README.md（如有）和 main / index 文件了解项目类型
3. grep -r "ADR\|PROPOSAL\|RETRO\|交接\|状态.md" 看有没有 framework 痕迹
4. 然后按 19 维度逐项判定

完成后输出报告。
````
