## 件 1 — F-205 · ErrorBoundary 防白屏

### What
2026-05-08 落地。给 5 个 Route 各包一个 React class ErrorBoundary，捕获运行时异常 → 落 deviceEvents.error 表 + 显示「咪咪打了个嗝」温暖回退卡。Profile 页加「最近的小问题」Card 展示最近 5 条。L3。5 个 vitest 单测覆盖 serializeErrorPayload。

### 顺手
- ErrorBoundary 是 React 经典 class component pattern，写起来无歧义
- 错误事件落到 deviceEvents 表（已存在）— 没动 schema，只约定了 payload 结构，省一个 ADR
- 文案"咪咪打了个嗝 🫧"对了产品语气

### 卡壳
- 真机故障注入 smoke 时发现：**5 个 ErrorBoundary 共享了一个组件实例**，导致 Health tab 崩了之后切 Home tab 仍然显示 fallback。原因：React 把同类型组件视为同一节点复用，state 不重置
- 修法：给每个 ErrorBoundary 显式加 `key="health"` 等独立 key，强制每个 Route 的 boundary 独立

### 学到
- React ErrorBoundary 的 state 复用是个隐藏坑，跨 Route 必须独立 key
- "三层测试 + smoke" 的价值：单测过 + build 过都不能保证真机不出问题，必须真机验证至少一次故障路径

### 教训
- 写 ErrorBoundary 时如果有多个 sibling 实例，**默认就要给独立 key**
- 故障注入测试要先想清楚边界场景（不只是"能否捕获"，还要"捕获后能否恢复 / 切到别的 tab 是否影响")

### 下次怎么办
- 给跨 Route 的 wrapper 组件做"是否需要独立 state"的设计决策清单
- ErrorBoundary 模板加注释：`// 必须配 key 区分不同 boundary`

## 件 2 — ADR-004 · 大文件拆分（PROP-001 B3 全拆）

### What
2026-05-08 落地。9 个源文件 ≥ 6500B（其中 5 个 > 9KB），mount 写入限制（~9105B）反复触发截断（已 2 次实际事故丢字节）。按 B3 全拆方案：13 个新小文件 + 5 个 features import 重写，所有源文件 < 6.7KB（最大 Home 6673B），后续 patch 又把 Home 降到 5647B。

### 顺手
- 拆分粒度按职责（HomeStatBar / HomeTaskSections / HomeHydration / HomeWorkout / HomeHabits / HomeAIBriefing / HomeChat）— 自然边界清晰，没有"拆得太碎"
- 全程 Python 写入（避免 Edit/Write 截断），分阶段提交（components → Home → Timeline → Health → Profile → Accounting → Home v2）每阶段独立验证
- import 替换全部用 sandbox 静态校验（括号平衡、末尾完整性、import 路径解析）— 比真机验证快 10×

### 卡壳
- Profile.jsx 在 import 替换时再次踩了 mount 8713B 截断坑（末尾丢 `);}` 收尾）— 已知约束第 1 项的"Edit 比 Write 更阴险"实例
- 拆完后发现 PROP-001 漏列 Accounting.jsx（8162B → import 后涨到 8288B），同日补拆
- Home.jsx 一直在 6KB 边缘（首次拆完 6673B），后又"彻底解决"再抽两个内嵌 Card 才到 5647B

### 学到
- mount 写入限制是项目最痛的物理约束，所有规则（"<6KB 用 Edit / 6-9KB 必须 Python / >9KB 拆"）都源于此
- "彻底解决"比"够用就行"更省时间 — 6673B 留在边缘以后随便加几行就破限，干脆抽到 5KB 区
- ADR + PROPOSAL 双轨记录的价值：PROP-001 是"咪咪提议给 zlbdh 决策"，ADR-004 是"决定 + 实施记录"，两者职能不重

### 教训
- 文件大小体检要全量扫描（不只 features，也含 shared、app、test），不然总有漏网
- 把"大文件红线"做成 rules/code/README.md 的"当前高危文件"段，每次修改后回头看一眼

### 下次怎么办
- 任何 .jsx 文件接近 6500B 时，主动提议拆分（不等到 9KB 才动）
- 写 commit 前先 `wc -c` 自动 check 一遍

## 件 3 — PROP-003 · 导航重构与聊天独立化

### What
2026-05-09 落地（同日完成，含 F-203 后串接）。底部 5 Tab 重排：今日 / 健康 / **聊天** / 记账 / 时光；「我」上提到右上角永久头像入口（PageHeader rightAction prop + ProfileAvatarLink 默认）。新建 `{{APP_REPO_DIR}}/src/features/chat/Chat.jsx`，把 HomeChat 全部交互平移过来 + 自动滚到底 + F-203 归档抽屉直接挂这里。L3。16 个新 vitest（chatInteraction + Chat），共 66 测试全过。

### 顺手
- PROP-003 文件本身写得很清晰（zlbdh 自己写的），照清单做就行，几乎零设计争议
- F-203 已经把 chatArchive.js 和 ChatHistoryDrawer.jsx 抽出来 — Chat.jsx 直接复用，不用重写
- ADR 编号冲突当时被立刻发现（005 已被 F-203 占）+ 双向确认（zlbdh Cowork 端同步改 PROP-003）— 流程没出岔子
- PageHeader 加 rightAction 用 default = `<ProfileAvatarLink />` + `null` 显式隐藏的 API 设计很自然，5 个 feature 几乎零改动就能用

### 卡壳
- 真机 smoke 时发现 Profile 页本身也显示了头像（self-link） — ADR-006 写了"Profile 页传 rightAction={null}"但实施 Profile.jsx 时漏了。当场 hotfix：加一行 `rightAction={null}` 到 Profile.jsx 的 PageHeader，重新出 APK 装机验证 OK
- 真机 adb 自动化的 tap 坐标估算反复出错（点歪了 2-3 次才点中目标）— 这是 adb + Capacitor WebView 的固有问题，不是项目代码问题

### 学到
- ADR 写得再细，实施时一定要拿 ADR 当 checklist 一条条对，不然 hotfix 频发
- 「我」上提到右上角是个低成本但高价值的信息架构调整 — 用户路径从 4 步→1 步，硬核收益
- ProfileAvatarLink 提取成共享组件后，Home 的自定义 hero header 也能直接复用 — 视觉一致性免费

### 教训
- 实施清单 = ADR 的"决定"段 + ADR 的"视觉规范"段，两段都要逐项划掉才能进 smoke
- 跨 feature 的迁移（HomeChat → Chat）要在迁移前 grep 一遍引用清单，确保移除时无残留

### 下次怎么办
- 对每个新 feature 的页面，开发完先问"PageHeader 的 rightAction 我传了什么？"
- 跨 feature 迁移前先列引用矩阵（被谁 import、谁的 props 传给它），降低漏改概率

## 横向观察 — 三件 L3+ 共有的 pattern

### ✅ 做得好的
1. **每件都有 ADR + PROPOSAL 双轨**：ADR 记决定 + 实施细节，PROPOSAL 记审批链路
2. **文档与代码同步**：每次 L3+ 都更新了 `Docs/3-开发文档/` 相关章节 + `需求历史.md`
3. **测试驱动**：纯函数抽出 + vitest，避免把所有逻辑塞 React 组件
4. **真机 smoke**：每件都做了真机验证，不只信任 vitest

### ⚠️ 反复踩的坑
1. **mount 写入限制**：F-203 + ADR-004 + PROP-003 三件都跟它打过交道；现在通过 < 6KB 软线 + Python 兜底基本控制住，但仍然每次新文件都要看大小
