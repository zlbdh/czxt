## 共性洞察 — 5 个

### 洞察 1：规范层改动比业务代码更难
- 业务代码改完跑测试就知道对不对
- 规范层改完没有「跑测试」机制——只有人后续踩坑才知道

应对：项目体检 + 状态推断 + RETRO 三层防御

### 洞察 2：mount 缓存是 Cowork 的工具限制，不是项目问题
- Edit/Write 走 Cowork 直连，立刻生效
- bash 通过 mount 读，可能滞后几分钟
- **不能修，只能绕**——所有 skill 都加「Read 优先」原则

### 洞察 3：「单一维护点」≠「单一目录名」
- ADR-003 当时为了维护点用 `rules/` 一个名涵盖一切 → 语义错位
- ADR-007 改成 `agent/` 顶层 + 5 子目录 → 两个目标同时达到
- **教训**：下次类似决策同时考虑「维护点 vs 语义」两维

### 洞察 4：「全自动」不是「无人值守」
- AI 是响应式的，必须有 prompt 触发
- 「90% 全自动」实际是「最大化降低用户提醒成本」
- 不要承诺做不到的「100%」

### 洞察 5：迭代次数 vs 边际收益
- v9 → v10 → v11 → v15：每轮发现 2-4 个真问题 → 高 ROI
- v15 → v20 → v22：每轮发现 6 个监控级改进 → 中 ROI
- v22 之后：可能再迭代 3 轮才发现 1 个真问题 → 低 ROI

**到稳态了**——再迭代不如反思。
