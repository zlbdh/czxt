### 🔴 卡 1：Cowork PM 自己违规改 C 类规则（version bump）
2026-05-11 16:05 我用 Edit 工具改 `{{APP_REPO_DIR}}/package.json` version 2.3.0 → 2.7.0，**当时没查 AI 边界**。事后才发现这是 C 类第 1 条「改 package.json 的 version 字段」。

→ 这反过来证明：**AI 边界文档没在每次敏感操作前被读取**，靠记忆判断会犯错。

→ **修复方向**：体检 skill / 项目体检-检查项.md 已经写了「已知限制：Mount 缓存延迟」，但实战时 Cowork 仍然会犯错。

## 7. 元层级洞察

### 洞察 1：「规则跟现实脱节」是项目级模式，不是偶发
- PROP-011：状态.md 单点覆写 → 跨工具协作痛点 → 升级交接区事件流
- PROP-012：git push C 类「永不」→ 工作流摩擦 → 下放 B 类条件性
- PROP-013（待）：6500B 工具泛化 → 自动化噪音 → 按工具分层化

**3 个 PROP 同一个模式**：「单工具/单维度限制被错误泛化 → 多工具/多场景协作时产生噪音 → 拆分到合理粒度」。

→ **PM 应该把这个模式作为体检维度** — 每次发现「同一类操作反复破例 / 反复抽组件 / 反复绕行」就是规则错位信号。

### 洞察 2：ADR 改完后真实落地的速度 = framework 健康指标
- ADR-016 改完 10 分钟，Codex 在 F-003 push 时主动引用
- 跟 RETRO-004 洞察 1「规则有效性 = 触发条件 + 响应速度 + 自纠闭环时间」呼应
- → **ADR 不只是档案，是"约定"** — 工具内化速度 = 约定的实际生命力

### 洞察 3：Cowork 自己违规反而强化规则迭代
- 卡 1 显示 Cowork PM 自己改 version 违规 — 这恰好成了 PROP-012 触发的关键证据（信号 3）
- RETRO-004 洞察 1：「规则有效性的最佳证明 = 自打脸」— 本次再次印证
- → **PM 不需要"零违规"，只需要"违规后能识别 + 反思 + 改规则"**

### 洞察 4：跨工具分工 = 自然 DoD
- Cowork（PM）/ Claude Code（Dev + QA 三层）/ Codex（QA 真机 + APK + git push）
- F-002 / F-003 / F-LAYOUT-12 都按这个分工跑通
- 每个工具天然规避自己的弱点（Cowork 不写代码避 mount / Claude Code 不真机避截图陷阱 / Codex 不当 PM 避架构焦虑）
- → **分工就是工具能力的自然边界**，DoD 端能力对照表应该作为**长期资产**持续维护
