## 3. 哪里卡（坑 / 低效 / 没必要的步骤）

### 🔴 卡 1：数据双源（profiles vs userProfile）
- 旧 `db.profiles` 表（含 height/weight/age/goal/bodyFat）+ 新 `db.userProfile` 表（id=1，5 段 + 人格模式）字段重叠
- 任务原文「在现有内容之前插入」按字面执行 → 用户「我」Tab 看到两个卡片，可能填两遍
- Claude Code F-PROFILE-12 / Codex F-PROFILE-12 push / Codex Sprint-2 收官 都主动标记
- → **应开 PROP-015 治理**（3 选 1：迁移 / 只读 / 合并）

### 🔴 卡 2：DEVIATION_KEYWORDS 简单 includes 误命中
- 当前用 `userMessage.includes('我能吃')` 判定破例
- 「我能吃饭吗？」（普通问题）也会触发破例模式
- Claude Code F-CHAT-23-DEVIATION 主动标记
- → **应改进检测精度**（正则边界 / 或 LLM 自判）

### 🟡 卡 3：3 个 ≥6500B 警戒文件长期治理
- `{{APP_REPO_DIR}}/src/app/useAppData.js` 6789B
- `{{APP_REPO_DIR}}/src/shared/database.js` 7084B  
- `{{APP_REPO_DIR}}/src/features/profile/ProfileSections.jsx` 7290B
- 按 PROP-013 规则 Cowork 视角软建议，Claude Code/Codex 视角 < 8KB 安全
- 不阻塞当下，但**继续扩这些文件前要先抽组件**

### 🟡 卡 4：状态新鲜度靠 prompt 纪律
- F-CHAT-3 实现「关键决策前重确认」全靠 system prompt 指令
- 没有结构化机制（如 sessionMemory 追踪过期决定）
- LLM 行为不稳定，可能漏触发
- → Sprint-3 可考虑加强机制

### 🟡 卡 5：LLM 行为测试只能真机 smoke
- vitest 211 测试只验「拼接正确性」（system prompt 含哪些字段）
- 真实行为（凌晨降级 / 三模式语气 / 破例检测）只能 ADB + 真机问咪咪验证
- 成本高 + 不可自动化
- → 可考虑「mock LLM 回放测试」

### 🟡 卡 6：PM 自纠
- Cowork 给 Claude Code 的提示词路径错（features/me vs features/profile）— PM 没先 grep 项目结构
- Cowork 自己之前改 `package.json` version 时违 C 类（PROP-012 信号 3 已记）
- → PM 应主动用刚加的 AI 边界硬提醒（敏感操作前 grep）

### 🟡 卡 7：记忆系统 4 个长期弱点（zlbdh 提问触发盘点）
- 没有"主题视图"（如 git 相关规则散在 3 个目录）
- 历史档案累积无 prune 机制（18 张交接卡，未来 100 张时？）
- 状态.md 顶部「已接手列表」会膨胀（18 张已显长）
- 没有"全局加载上限保护"（用户「读所有 ADR」会冲击上下文）
- → 都不阻塞业务，可在 Sprint-3 后开 PROP-015~018

- v3.0 PRD「灵活性引擎」+「系统状态感知」核心原则都是
- → **可能比单个 feature 重要** — 这是 zlbdh 反复强调的元原则（"不能用户说啥同意啥"）
