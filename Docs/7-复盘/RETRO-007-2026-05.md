## 7. 元层级洞察

### 🌟 洞察 1：「前置治理债」是业务速度起飞的核心
- PROP-015 数据双源治理 = Sprint-3 启动前 0.5 天投入
- 红利贯穿 Sprint-3 整 6 个 features（5 业务 + 1 PROP）
- → **「业务前置阻塞」类 PROP 应该当一等公民处理，不当 backlog**

### 🌟 洞察 2：质量门禁回退 ≠ 失败，是 framework 自检力的证明
- F-PLAN-1 smoke 阻塞 → 不硬过 → 回退修复 → 复跑通过
- 多花 30 分钟，**但避免了"看起来完成实际有 bug"流出 prod**
- → **下次类似情况不慌，反而要鼓励 Codex「敢说不通过」**

### 🌟 洞察 3：Claude Code 主动为 PM 减负（议题清单整理）
- F-CHAT-4 交接卡 ⑥ Claude Code 主动整理 RETRO-007 候选 19 条议题
- PM 写本 RETRO 时直接复用 — **节省 PM 议题盘点时间 ~30%**
- → **跨工具协作的真正含义不是 PM 单方面"管理"，是双方互为输入**

### 🌟 洞察 4：RETRO 完成率下降 = 业务挤压 framework 演进信号
- RETRO-006 完成率 100% / RETRO-007 完成率 33%
- 原因：Sprint-3 全塞业务 features，没给 framework 演进留时间
- → **Sprint 规划要主动给 framework 演进留 1-2 天「呼吸窗口」**

### 🌟 洞察 5：PM 自纠次数 4 → 6 = PM 提示词精度仍待提升
6 次自纠分布：
- 路径错（features/me vs features/profile）
- C 类违规（version bump）
- 过度规则化（chat 简版给 Owner）
- grep 漏扫 5 个调用方
- Home 三目标位置错位
- 空态实现机制没说清

→ **共性**：PM 写指令时**对项目实际状态的 grep / Read 验证不够**，靠记忆判断。
→ **修复**：写 PROP / 提示词前先 grep / Read 实际代码（已加到 RETRO-005 卡 1 修复，PROP-014 简版自示范案例已生效，但 grep 习惯还没养成）

### 🌟 洞察 6：测试密度爆发是质量基线抬升的最直观指标
- Sprint-1: 96 → Sprint-2: 211 (+115) → Sprint-3: 399 (+188)
- 平均每 feature 加 ~15-20 个测试（Sprint-2 平均 16 / Sprint-3 平均 31）
- → **Sprint-3 测试密度翻倍** — 跟 PROP-013（按工具分层）+ PROP-014（chat 简版强制）的"工具自律"环境呼应
