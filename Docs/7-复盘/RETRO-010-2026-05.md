## 🎓 学到的事

### 教训 1 — AC 实证驱动 > PM 主观拍数值

F-NIGHT-1 v1 失败 = PM 主观拍「< 50 字」无实证基础。教训：PRD AC 涉及数值硬约束（字数 / 时长 / 频率）时**避免单一硬约束**，改用**双约束（结构硬 + 数值软 + 运行时后处理）**。AC 起稿前可考虑加「数值由真机 smoke 实证后调整」字段。

### 教训 2 — PM 不默认推测前必先核实 v3.0

议题 BR（AI 模型）+ BP（Live LLM 401）+ Sprint-6 F-DIARY-1 UI 方案 + ship 卡分支名 master = **同模式连续 4 次**。已积累 3 条 feedback memory：
- `feedback_pm_no_default_inference.md`（信息盲区"我不确定"）
- `feedback_pm_verify_ui_first.md`（UI 架构必先 grep verify）
- 议题 CD 候选：PM 起 ship 卡前必走 verify checklist（git branch / 模块入口 / Tab 配置）

### 教训 3 — LLM thinking model 与 max_tokens 边界

MiMo (thinking model) 在 max_tokens=150 + 连续多轮会空回复 — 这跟非 thinking model 行为完全不同。教训：`max_tokens` **不能当字数工具用**，字数约束改用运行时后处理（normalizeNightReply 模式）。Codex 真机实证发现，PM 没在 PRD AC 设计阶段识别。议题 CE 候选：建立"模型类型 vs prompt 设计模式"映射表。
