## 📚 教训定式（3 条，候选元规则池）

### 1. #95 bare-word-in-comment 本批 3 撞 → 定式化（🔴 提元规则池）

v3.19（`/normalizeEvent/` 撞注释「绕 normalizeEvent」）、v3.20（同型）、v3.22（`/查看食谱/` 撞 Modal 自述注释「不拷 F-F2/F-F4（查看食谱…」）。三撞同根：**source-grep 禁词锁了会出现在说明注释里的裸词**。

**定式**：source-grep 禁词断言**必锁代码标识** —— 带括号的调用 `fn\(`、组件名 `RecipeDetailModal`、import 路径；**绝不锁**会进注释/文档的中文裸词或无括号函数名。（v3.22 实施者已自抓自修并写成注释定式，本 RETRO 提为元规则候选，沉淀 PM 排编号。）

### 2. v3.20 真机阻塞案例：单测全绿 ≠ LLM 链路通（🔴 候选）

weekly 第二刀 1237 单测全绿，真机「每周三复诊」却走普通聊天。双因：①prompt 规则①「有时间的临时安排」语义门把周期事项判 isEvent:false ②normalizeEvent `if(!date||!startTime)return null` 静默吞无锚点周期（monthly 靠「号」侥幸过、weekly 无 day-of-month 必死）。**根因的根因：LLM 行为层没有确定性测试** —— 纯函数测试覆盖不了「LLM 输出形态→normalize→出卡」链路。

**定式**：LLM 链路每加新形态（新 freq/新场景/新字段），必须加「**LLM 输出 mock→normalize→可出卡**」确定性链路测（v3.20 修复时已补：normalizeEvent 兜底 + mock 锁，weekly/yearly 解析失败从此在 vitest 层拦住）。修复模式 = **双层 belt**（prompt 开门 + normalize 兜底只补缺失）+ 三重 0 回归结构保证。

### 3. Cowork mount stale/截断高频爆发 → 防御打法沉淀（ADR-025 补录候选）

本 session ~8 次实例：Profile 7507↔7850 / EventCard 6217↔8059 / git diff 假截断（`isEventActiveOn` 显示 `return fal` 断词而该函数根本没改）/ defaults.js diff 空显 / index.js「2M vs 3 改」假差异 / 状态.md 尾部冻结+行号漂移+读缓存延迟回显。

**防御打法（实战验证）**：
1. 读**已提交**内容一律 `git show HEAD:<path>`（git object store 绕缓存/截断）；
2. **git status/diff 元数据可信**；`wc -c`/Read 对**本 session 读过的文件不可信**（stale），对**新建文件可信**（fresh）；
3. 状态.md 这类追加型文件**只用 `>>` O_APPEND**（落真实 EOF、绝不 clobber），**绝不 read-modify-write**（按 stale 写回=抹掉别人的行）；追加后回显延迟是常态、从未丢行，**不要重复 append**；
4. 文件完整性**终证 = 实施者/Codex 真 shell 的 build+vitest**（截断=语法错=build 必挂），Cowork 只做元数据+实现段核验并诚实标注边界。
