# RETRO-021 · PROP-044 B-lite framework 受控并行首次试点

> 时间窗：2026-06-14 · 触发：PROP-044 AC3（真实大批量 framework 批次试点 worker 并行）+ AC4（试点后评估是否升 ADR）· 类型：framework 治理专项（非 Sprint 批次）

## 一、这批改动做了什么（对照 PROP-044 AC）

- **前序**（已落地）：PROP-044 zlbdh approve **B-lite 受控并行** → `子agent调度机制.md` §三.5 + `角色边界.md` + 派工卡「写集互斥声明」（AC1/AC2）。
- **AC3 试点**：项目 PM 派 **3 个真实并行 worker**，给 `能力资产/` 下 **26 个缺 scope frontmatter 的 .md** 加 frontmatter（PROP-037 / ADR-028 真债）：
  - worker A = `rules/`（11 文件 / project·semantic·on-demand）
  - worker B = `skills/`（9 文件 / project·procedural·on-demand，1 个 triggered）
  - worker C = `shared/+tools/+mcp/`（6 文件 / project·semantic·on-demand）
  - 写集按子目录**互斥**切分，3 worker **同一条消息并发**。
- **主会话单点合并 + 体检 gate**：独立验证（精确 26 文件变更 + 禁并行哨兵未碰 + 抽检正文完整）+ `check-operating-system.ps1` **exit 0**（P4a–P4h 全过）→ 接纳。
- **AC4** = 本 RETRO。

## 二、哪里顺（保留）

1. **写集按子目录互斥 → 0 竞态 0 越界**。独立验证铁证：全框架仍缺 frontmatter 数从 61 精确降到 35（=61−26），证明无界外写入；禁并行哨兵 `角色边界.md`/`CHANGELOG.md` 首行仍 `#`（worker 没碰）。
2. **派工卡「写集互斥声明」+ 显式禁止清单（含单源单点文件）有效**。3 worker 全部 0 触碰 `状态.md`/`交接区`/`CHANGELOG`/`元规则池`/`角色边界`。AC2 新字段经实战验证。
3. **brief 给方向不锁手段 → worker 产出更优**：worker A 自主用 Edit 替代 Write（更稳保正文），worker B 自主判 `状态推断-跨session监控` 用 `loaded: triggered`。证明 worker 有判断空间能改进 brief 的建议做法。
4. **体检 gate 作为 framework 写入接纳闸有效**：P4f 轨迹 / P4g 计数 / P4h 锚点全自动核，exit 0 才接纳，0 false-negative。
5. **独立越界探针廉价强力**：「全框架仍缺总数 == 原数 − 派工数」一条命令即证无界外写入。

## 三、哪里卡（坑 / 低效）

1. 🔴 **barrier 浪费**：worker 时长 A=79s / B=305s / C=55s，并发 wall-clock = max ≈ **305s**。A、C 早在 79s/55s 完工却空等到 305s（~226s / ~250s 闲置）。写集按**文件数**切（11/9/6）但 skills 文件更大更需判断 → B 是瓶颈。**教训：framework 受控并行派工应按"预估工作量"均衡，不只文件数；或用 pipeline 去 barrier。**
2. 🟡 **验证成本真实存在**：主会话独立验证（scan + 抽检 + 体检）≈ 2-3 步。本次 26 文件划算；但小批量（6-8 文件）验证开销可能吃掉并行收益 → 印证 §三.5「≥6 文件」阈值合理，甚至可上调。
3. 🟡 **cosmetic**：部分文件 frontmatter 后出现双空行（worker prepend + 原有空行）。无害（markdown 折叠），下次 brief 可提示规整。

## 四、要不要改规范 / 升 ADR（action item）

**升 ADR 建议：暂不单升，标 ⭐ ADR 候选 / 再 1-2 次实战后升。** 理由：

1. 元规则升级路径要求"累积 3+ 同模式 + RETRO 评估"——本次是 **首次**试点，仅 1 个数据点；
2. B-lite 已是 §三.5 **现行规则**（PROP-044 approve），不升 ADR 也已生效、有约束力；
3. 升 ADR = 锁宪法级，应等机制在 **2-3 种不同 framework 批次**（不只 frontmatter backfill，如批量链接修复 / 批量 playbook 增补）上验证通用性。

**候选元规则（下次实战验证项 / 走 PROP 才落地，本 RETRO 只观察）**：

| 候选 | 内容 | 出处 |
|---|---|---|
| **DU** | framework 受控并行派工按**预估工作量**均衡（非文件数），或 pipeline 去 barrier | 本次 barrier 浪费实证（B 305s 拖死 A/C）|
| **DV** | 受控并行验收加「变更文件数 == 派工总数」廉价越界探针，考虑进 §三.5 验收 checklist 或体检 P4i | 本次 61−26 精确核实证 |
| 观察 | §三.5「≥6 文件」阈值合理，验证成本数据支持，1 次后暂不调 | 本次验证成本观察 |

**触发升 ADR 条件**：再 ≥1-2 次不同类型 framework 批次受控并行成功 + DU/DV 候选累积 → 那时起 PROP 升 ADR（续 ADR-027 子 agent 调度，或新 ADR）。
