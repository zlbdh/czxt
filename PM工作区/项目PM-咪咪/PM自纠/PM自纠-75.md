# PM 自纠 #75 — 项目根 tools/ vs 能力资产/tools/ 命名冲突

> **时间**：2026-05-22 15:30
> **触发**：zlbdh 反问「{{PROJECT_ROOT}}\能力资产\tools、{{PROJECT_ROOT}}\tools；为什么有两个 tools？」
> **议题**：议题 CM v3.2（命名跟不上设计 / 第 3 次同模式累积）
> **同模式累积**：PM 自纠 #60（PM记忆中枢→PM工作区）+ PM 自纠 #61（角色定义文件名 vs 工作目录名）+ **本次 #75**（顶级目录命名冲突）

## 一、错向

我在 **task #90-#92**（2026-05-22 凌晨 / PROP-027 v2 改进 2 完整版落地）新建了 `{{PROJECT_ROOT}}\tools/` 作为 PowerShell 脚本目录（check-operating-system.ps1 + check-pm-tracking.ps1），**没检查**是否已有同名 `能力资产/tools/`。

实际上 `能力资产/tools/` 早在 **2026-05-21 PROP-028** 就建立了（构建脚本.md + 依赖矩阵.md 等 markdown 声明文档）。

两个目录性质完全不同：
- 项目根 `tools/` = **可执行 PowerShell 真代码**
- 能力资产 `tools/` = **声明式 markdown 文档**

但都叫 `tools/` → 二义性命名冲突。

---

## 二、根因

### 1. 新建顶级目录前未 `ls -d` 检查冲突

我建 `tools/` 时直接 `mkdir tools && ...`，**没先 `ls 能力资产/tools/`** 也没 grep "tools" 找已有目录。

### 2. 沉淀 PM「沉淀者」框架体检未覆盖命名冲突检测

task #102-#108 沉淀 PM 多次做 framework 体检 + 角色边界审核，但**只查文件存在性**（议题 BE 起手必查），**没查顶级目录命名冲突**。

### 3. 议题 CM 治理未覆盖"顶级目录"层级

- PM 自纠 #60 治的是"目录名跟不上设计意图"（PM记忆中枢→PM工作区）
- PM 自纠 #61 治的是"文件名 vs 目录名不一致"
- 但议题 CM 没显式治"顶级目录命名冲突 / 二义性"

### 4. 没参考行业惯例

PowerShell 脚本目录业内惯例叫 `scripts/`（npm scripts / shell scripts / GitHub Actions），而 `tools/` 更常用于"工具治理"概念。我用 `tools/` 装可执行脚本是命名歧义。

## 五、永久学习

### 新建顶级目录前必查清单（PM 自纠 #75 → 沉淀 PM 速查表）

```
1. ls -d <target_dir>              # 检查同名直接冲突
2. find . -type d -name "<name>"   # 检查全工程同名目录
3. grep -r "<name>" --include="*.md" | head  # 检查 README 引用
4. 业内惯例对照（scripts/ vs tools/ vs lib/ vs bin/ ...）
5. 命名传达"意图"还是"形式"（PM 自纠 #60 教训）
```
