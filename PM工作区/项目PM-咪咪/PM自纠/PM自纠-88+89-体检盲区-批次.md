## #88 — PM-产品经理.md 误判残留（PROP-042 启动时发现）

**错向**:体检报告判 `PM-产品经理.md` 为"该删的历史残留",建议"1 分钟零风险删"。

**真相**(对比新版后发现):
- 新版 `产品PM-需求拆解者.md` 开头明写:"**吸收并替代 PM-产品经理.md(旧文件保留作历史档案,PROP-004 / ADR-007 不动历史档案铁律)**"
- `项目PM-咪咪.md` 也注明:"保留旧文件作历史档案,**不删**"
- 内容已完整迁移(工作步骤/示例对话/F-024 都迁了)

→ **它不是残留 / 是有意保留的历史档案**(ADR-007 铁律)。

**根因**:体检只看"文件名是否旧"+"是否被新版取代",**没看新版文件内"保留说明"**。差点违反 ADR-007 历史档案铁律 + 断 3 个活跃引用。

**修复**:撤回"删"建议 / 保持现状。

## #89 — 体检字节数盲区(PROP-042 第 2-3 棒撤回时发现)

**错向**:体检按字节 >8KB 标 7 个"红区文件",建议拆分。基于此起 PROP-042 列 5 棒清单。

**真相**(逐文件 Read 真核后发现):
- 第 1 棒 `decision-checkpoint`(18KB) → 有大段"案例/历史"长尾 → **真能拆** ✅(已 task #117 完成 / 减 ~5KB)
- 第 2 棒 `INDEX`(14.8KB) → 6 节都是核心 / 唯一"历史"段(第六节)只 728B → **拆不出量**
- 第 3 棒 `角色边界`(13.6KB) → task #106.4 已拆过 / 剩下全是核心 9 PM 矩阵 → **不该再拆**
- 第 4-5 棒待评估

**根因**:体检盲区 — 只看"字节数",没看"结构性可拆 vs 必要核心"。"红区"判定需要分两类:
1. **可拆肥**:有"案例/历史/长尾段" 与核心混合(如 decision-checkpoint)
2. **结构性肥**:全是核心内容 / 已无可拆空间(如 INDEX / 角色边界)

强拆第 2 类 = 损害高频体验 / 不偿失。

**修复**:PROP-042 第 2-3 棒撤回 / 第 4-5 棒列 backlog 等真有结构性可拆机会再启动。

## 沉淀(议题 DN candidate / 体检质量改进)

体检 SOP 升级:**>8KB 标红 + 必跟"可拆性 vs 核心性"判定**

1. 先 `awk` 出段字节分布
2. 评估每段是"案例/历史长尾"还是"必要核心"
3. 只对"长尾段 ≥30% 文件大小"建议拆
4. 历史档案豁免必看文件内"保留说明"(ADR-007 / 不只看文件名)

议题 DN candidate「体检质量 SOP — 可拆性判定」→ 沉淀 PM 速查表加规则。
