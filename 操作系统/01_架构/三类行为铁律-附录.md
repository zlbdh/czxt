## 灰区例子

没明文规定时默认 ask，尤其是下面这些看似小、实际可能越界的动作：

- 改 README.md 里的项目介绍措辞：先判断是产品定位变化（B 类）还是 typo（A 类）。
- 改 `.gitignore`：一般可逆，但如果影响 build 输出、密钥保护或发布链路，先按 B 类确认。
- 改既有测试用例：如果是 PRD 变化后的契约同步可做；如果只是测试挡路，先停下来解释。

## A 类完整例子

| 行为 | 备注 |
|---|---|
| L1 改动（文案 / typo / 单点样式） | 可直接做；framework 路径留 `状态.md` / CHANGELOG 轻量轨迹，不需要 PROP / 完整 handoff |
| 写 PROP 到 `确认改动/待审批/` | AI 提议本身不需要审批 |
| 跑测试（vitest / esbuild / vite build） | A 类表示无需审批；仍遵守角色边界，发布复核归测试发布 PM |
| 读现有代码 / 文档 | 任何时候 |
| 在 sandbox（`/tmp/mimi-test`）里建/改文件 | 不影响真实代码 |
| 写 / 更新开发文档（`Docs/3-开发文档/`） | 代码改了就同步改文档 |
| 写测试代码（`*.test.js`） | 不需要审批 |
| 写 task list、更新 task 状态 | 自我管理 |
| 在 sandbox 写代码草稿（用于 demo / 验证） | 真要进 `src/` 才需要按等级走 |
| 写 / 更新知识库（按角色白名单：如 `Docs/3` / `Docs/7` framework，`Docs/1` 需求） | 把代码事实沉淀下来，不越过路径白名单 |

## 撤销机制

咪咪做了 A 类操作后，zlbdh 反悔：

- **代码层面**：看 git diff，用最小补丁还原到改前状态。
- **文档层面**：手动改回或在明确授权时 revert 对应 commit。
- **预防层面**：每次 A 类操作完，汇报里说清楚"我改了 X，diff 在这里"，让 zlbdh 能立刻反悔。

不要把 A 类做成"既成事实"。

---

## 原则总结

> **可逆 + 沙箱 = 自动；不可逆 + 真实 = 必须 ask。**

- 沙箱里随便玩。
- 文档级事实沉淀可以自动做，但 framework 路径要留轨迹。
- 代码 / 配置 / 发布 / 数据越接近真实用户和外部分发，越保守。

---

## 跟 L1-L4 改动等级的对应关系

AI 边界（A/B/C）跟 L1-L4 改动等级是正交但有重叠的两个体系：前者管「该不该做」，后者管「怎么做」。

| L 等级 | 例子 | A/B/C 边界 | 为什么 |
|---|---|---|---|
| **L1** 文案 / typo / 单点样式 | 改副标题 / 改一处文案 | **A 类（自动）** | 可逆，直接做；framework 路径留轻量轨迹即可 |
| **L2** 行为微调 / 单点 bug | 修一个功能行为 / 加错误提示 | **A 类（自动）** | 可逆 + 局部修复，AI 直接做 |
| **L3** 新功能 / 跨多文件 | 加 feature / 跨模块改动 | **B 类（必问）** | 新功能，zlbdh 必 review PRD 才开干 |
| **L4** 架构 / 数据 schema 变化 | 加新表 / 改 schema / 拆模块 / 加原生插件 | **B 类（必问）** | 架构变化，PRD + ADR 双审，等 zlbdh 决策 |
