## ⚠️ 敏感操作前必查 C 类清单（RETRO-005 卡 1 修复）

**任何**涉及以下范围的操作前，先 grep 本文件 C 类表确认；不要靠记忆判断。

涉及范围：
- 🔴 git（commit / push / tag / rebase / branch / remote）
- 🔴 `package.json` / `capacitor.config.*` / `vite.config.*` / `.gitignore` / `build-apk.bat`
- 🔴 API key / baseUrl / 任何密钥配置
- 🔴 用户数据 / IndexedDB / 真机数据 / `apk/` 历史 APK
- 🔴 `Docs/6-历史归档/` 历史归档
- 🔴 `_framework`-级文件（操作系统/02_智能体/* / 能力资产/rules/* / 操作系统/07_完整工作流/* / 能力资产/skills/* 等元规则）
- ⚠️ **B 类例外**：项目根 `AGENTS.md` 和 `README.md` 是 AI 入口文件，属 B 类（zlbdh review，不走 PROP/ADR），不在 C 类。

执行流程：
1. 拟动手前 → grep 本文件「### ❌ C 类」段，看动作是否被列入
2. 如果中了 C 类 → 立刻停手，不执行，也不把它当成 ask 后可通过项；若要改变规则先走 PROP/ADR，真实密钥外传、写 tracked 密钥、删除用户数据、伪造数据仍不可执行
3. 如果在 B 类 → 检查条件是否满足
4. 如果在 A 类 → 自动做，但事后报告 diff

## 三类行为

### ✅ A 类 — 咪咪自动做（不用问）

A 类原则：可逆、局部、非敏感、可通过 diff 解释。常见动作：读文件、跑测试、L1/L2 小修、写测试/开发文档/待审批 PROP、更新任务状态、在 sandbox 写草稿。

完整例子表见 [`三类行为铁律-附录.md`](三类行为铁律-附录.md)。

### ⚠️ B 类 — 必须先 ask zlbdh

| 行为 | 备注 |
|---|---|
| L3 / L4 改动开工 | PRD 写完后**必须等 zlbdh 说"开干"** |
| 改产品方向 / 删需求 | 影响 zlbdh 心智模型，必须确认 |
| 改 PRD 里已定稿的核心决策 | 重大方向调整 |
| 删除已存在的功能 / schema 表 / 字段定义 | 仅限设计层或迁移方案评审；真实用户数据擦除仍属 C 类 |
| 引入新依赖（npm install xxx） | 可能改包大小 / 引入风险 |
| 改 ADR（已写过的） | 历史决策不轻易翻案 |
| 改 build-apk.bat / capacitor.config.json / vite.config.js | 影响发布链路 |
| 改 `_framework`-级（这套规范）本身 | 元层级影响所有项目 |
| 把代码改动从 sandbox 推到真实 `src/` | 跨过沙箱-真实分界线 |
| **git commit / push**（main 常规操作，ADR-016）| 必满足 6 条件：① 仅 `{{APP_REPO_DIR}}/` main ② commit msg 真实 ③ 不 force/rebase/rewrite ④ push 失败停手不重试 ⑤ 交接卡明示 hash+push 结果 ⑥ contextual 授权 |
| **打常规版本 tag / push tag** | 仅允许绑定测试发布 PM「闭环者」发布闭环；需代码、测试、build、APK、smoke、版本号、交接卡一致，且不得删除/改写/移动已有 tag |
| **改 `package.json` version 字段**（PROP-012 / ADR-016）| 配套 APK 发版任务时可执行；单独 bump 仍需 ask |
| **改项目根 `AGENTS.md`**（ADR-021）| AI 入口文件，zlbdh review approve 后生效；遵循极简入口，详细规则在 `操作系统/` 和 `能力资产/` |
| 改项目根 `README.md` 核心定位（非 typo） | 影响产品介绍措辞 |
| **改 AI 配置默认值**（`{{APP_REPO_DIR}}/.env.local` baseUrl / model / apiKey）| **从 C 类下放 B 类**；必满足 6 护栏：`.env.local` gitignore、key 不进 tracked、DEV gate、user 配置优先、APK 不传第三方、不上 Google Play |

**Ask 模板**：直接对话问；问前先汇总已知信息，让 zlbdh 不用重新查上下文。

### ❌ C 类 — 永远不能动（PM 不主动执行 / 不主动问成可执行）

| 行为 | 为什么 |
|---|---|
| `git push --force` / force push / `git rebase` 已 push 的 commit | 重写历史不可逆，影响协作者 |
| 删除分支 / 删除 tag / 操作其他 fork-remote | 不可逆 / 越界 |
| 删除 / 改写 / 移动已有 git tag / 推 GitHub Release | 不可逆或面向外部分发；常规版本 tag 见 B 类闭环条件 |
| 上传 / 推送任何 APK 到分发渠道 | 发布给"谁"是 zlbdh 决定 |
| ~~改 AI 配置 baseUrl / apiKey 默认值~~ | **已下放 B 类**（议题 V / ADR-022 决定 2，2026-05-13，6 护栏锁定）|
| 删除 zlbdh 写的笔记 / 备份 / 用户数据 | 数据是 zlbdh 的 |
| 改 `apk/` 下已存在的历史 APK 文件 | 历史归档不动 |
| 改 `Docs/6-历史归档/` 已归档的内容 | 历史就是历史 |
| 任何"为了让 demo 看起来好"伪造数据的事 | 诚实第一 |
| 多 agent **并发写同一 framework 文件**，或并行写禁并行清单（`状态.md` / `交接区` / `CHANGELOG` / `元规则池.md` / `角色边界.md` / [`子agent调度机制.md`](子agent调度机制.md)）| 无 worktree 兜底 + 单 append 点；B-lite 只许并行写**互斥的非清单**文件 |

## 灰区速断
