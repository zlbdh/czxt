# Framework 演化哲学（双 PM 共享 — 加角色 / 扩 framework 前必读）

> 触发场景：zlbdh 或任何 PM 想加新角色 / 拆 framework / 扩 PM 子类时，**先 read 本文件做判断**。
> 历史触发：2026-05-19 zlbdh raise「我们是否需要模仿现实中公司的架构」→ PM 明确推荐**不模仿**，按"实际工作流约束"演化。
> 类比借鉴：议题 AJ 教训（PROP-020 路径 C 因方向不清被废弃）+ 议题 AY 教训（framework 复杂度也是债，不只代码债）。

## 一、核心原则：**不模仿现实公司架构**

现实公司架构（CEO / CTO / CMO / CFO / COO / VP / Director / Manager / IC）服务 5 项约束 — **AI 协作项目里多数不成立或反向有害**：

| # | 现实公司约束 | 在我们项目里 |
|---|---|---|
| 1 | 人的精力边界（一人不能干多领域）| ❌ AI 角色无精力上限 |
| 2 | 信息隔离（部门墙 / KPI / 政治）| ❌ 信息全在 markdown 文件 |
| 3 | 法律责任分担（合规需明确责任人）| ❌ 个人项目 zlbdh 一人决策 |
| 4 | 沟通成本（管理跨度 7±2）| ❌ 文件 + 交接卡 = 0 翻译损失 |
| 5 | 招聘效率（专业化分工便于雇佣）| ❌ 无招聘 |

照搬反例 — 大部分角色空转 + 引入"层级 / 部门墙 / 角色摩擦" = framework 债爆炸（比业务代码债更难拆，因为 markdown 角色文件无 split 路径）。

## 二、应该按什么逻辑切角色？

**按「思维模式 + 工作流约束 + 路径白名单」切角色，不按"职位 / 部门"切**。

每个角色 = 一种思维模式 + 一组路径白名单 + 一套起手流程。**同一个咪咪在不同时刻的视角切换**，不是"X 个员工"。

当前 9 PM 的真实切分逻辑：

```
项目 PM 咪咪 = 编排者视角（全局）
   ├─ 沉淀 PM = 元规则 / 记忆 / 复盘沉淀视角（元-层）
   ├─ 操作系统 PM = framework 维护者视角（操作系统/ + 能力资产/ + Docs/3+7 + 确认改动 + 交接区）
   ├─ 产品 PM = 需求拆解者视角（Docs/1-需求文档/ + 确认改动/待审批/）
   ├─ 技术 PM = 修复决策者视角（只读 {{APP_REPO_DIR}}/）
   ├─ 测试 PM = 质量门户视角（全部只读 / 验收策略）
   ├─ 运营 PM = 商业增长视角（向外另一个工作流）
   ├─ 开发 PM = 业务代码实施视角（worker 载体可换）
   └─ 测试发布 PM = 发布闭环 / push / APK / smoke 视角（发布主控载体可换）
```

## 三、加新角色的扩编三问

每次想新增 PM 角色前必跑（任何一问 NO → 不加）。这是**扩编专用**判断，不替代日常切角色的 `decision-checkpoint` Q1-Q7。

| Q | 判断标准 | 严格度 |
|---|---|---|
| Q1 实战痛点 | 现在有反复出现的工作流我搞不定 / 边界不清 / 错向？| 必有 ≥ 3 次实证 |
| Q2 频次 | 这个工作流一周会跑几次？ | ≥ 1 次/周（每月 < 4 次不加）|
| Q3 替代方案 | 现有角色用 sub-mode（议题 P 子分支模式）能否覆盖？ | 不能 |

扩编三问全过 → 起 PROP 设计新角色。任一不过 → 不加，议题 backlog 记下等高频再评估。

## 四、加角色不推荐方向 vs 推荐方向

### ❌ 不推荐（公司架构思维）
- 按职位（CEO / CTO / CMO / HR / 法务 / 行政）
- 按部门（产品部 / 技术部 / 市场部 / 销售部）
- 按层级（VP / Director / Manager / IC）

### ✅ 推荐（工作流约束思维）
- 按反复出现的**思维模式切换**（编排 / 维护 / 拆解 / 修复 / 验收）
- 按**路径白名单分割**（不同角色守不同目录）
- 按**外部触发条件**（A 阶段才有的用户支持 / 数据分析 / 商业化）

## 五、当前评估 + 未来演化指引

### 当前现状（2026-06-14）
- **角色数 9**（主 1 + 元-层 1 + 决策 5 + 实施 2）— v4.0 终态，继续收敛而不是扩编
- 议题 AJ 实战 30+ 次 0 越界 ✅ 必要性已验证
- ADR-038 已把执行能力从“再加 PM”改为“项目 PM 统一调度 worker/explorer 子 agent”

### 短期不加新 PM 角色
若只是需要并行检查、受控写入或验证，优先按 `子agent调度机制.md` 开 worker/explorer；只有新工作流稳定高频且现有 9 PM 无法表达时，才回到本文件的扩编三问。

### 未来候选（A 阶段公开发布后才考虑 — 都来自工作流约束）

| 候选角色 | 触发条件 | 替代方案（首选）|
|---|---|---|
| 客服 PM | 公开发布后用户找你修 bug / 答问题 | 项目 PM + 技术 PM sub-mode |
| 数据 PM | 公开发布后有 DAU / 留存数据 | 运营咪咪 sub-mode |
| 商业化 PM | 决定订阅 / 付费模型时 | zlbdh 直接决策 + 项目 PM 拆 Sprint |

**核心思路**：能用 sub-mode 覆盖的不加新角色；只在新工作流**稳定高频**且 sub-mode 不够时才加。

## 六、议题 BJ 候选（本文件元规则升级条件）

如未来发现本哲学有结构性盲区（例如某次加角色后 6 个月内业务 / 增长出现错向）→ 起 PROP-N 修本文件 + 升 议题 BJ ADR-N 永久化。

当前为 v0（仅 zlbdh 战略层一次性沉淀，未经长期实战）— 谨慎使用，遇争议优先 zlbdh 仲裁。

## 七、不要做（硬反例）

- ❌ **未跑扩编三问就加角色**（议题 AJ 教训）
- ❌ **按公司架构思维定义角色**（CEO / CMO 那种角色会空转）
- ❌ **加角色不写 playbook + 路径白名单**（PROP-020 路径 C 翻车样本 — 角色不可见就是不存在）
- ❌ **角色间设"汇报关系" / "层级"**（咱们是分支不是层级；冲突统一走 zlbdh 仲裁）
- ❌ **不在乎 framework 复杂度**（议题 AY 教训扩到 framework 层）
