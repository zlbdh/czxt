# 任务状态机（O-1 / PROP-029 v2 落地 / 2026-05-21）

> **🎯 5+2 态精简版**（5 个主状态 + 2 个分支态；适配{{PROJECT_NAME}}单人多 PM 职责流）

---

## 一、5+2 态定义

```
PLANNED        → 已起 PRD / handoff 卡，等接手
IN_PROGRESS    → 接手 PM 动手中（开发 PM / 测试发布 PM / 其他 PM；工具仅作执行载体）
DELIVERED      → 完成卡 + 单测/build/smoke 过 / 等 PM accept
ACCEPTED       → PM verify 通过 / 进入收档
CONDITIONAL    → 主体通过但需补证据 / 偏慢 / 待 verify
REJECT         → smoke fail / 回炉重做
CLOSED         → 完成 + RETRO 归档
```

5 个主状态 + 2 个分支态（CONDITIONAL / REJECT）= 7 个枚举。

## 二、状态流转图

```
PLANNED → IN_PROGRESS → DELIVERED ─┬─→ ACCEPTED → CLOSED
                                    ├─→ CONDITIONAL → IN_PROGRESS（补证据后）
                                    └─→ REJECT → IN_PROGRESS（v2 修复）
```

## 三、各状态触发条件

| 状态 | 触发 | 退出条件 |
|---|---|---|
| **PLANNED** | PM 起 PRD / handoff 卡 | 接手者读卡 + 开干 |
| **IN_PROGRESS** | 接手者按卡执行 | 完成卡写完 |
| **DELIVERED** | 完成卡 + 测试/build pass | PM verify |
| **ACCEPTED** | PM 验收通过 | 收档归档 |
| **CONDITIONAL** | 主体过 / 有小问题待验证（如议题 CF F-NIGHT OFF 偏慢） | 补证据 → DELIVERED |
| **REJECT** | smoke fail / AC 不过（如 W-3 v1 / F-NIGHT-1 v1） | 回炉 v2 → IN_PROGRESS |
| **CLOSED** | RETRO 归档 + 议题永久关闭 / Task completed | 终态 |

## 五、为什么不要 yuanxing 10 态

- yuanxing 10 态（含 WAIT_DECISION / NEED_MORE_EVIDENCE 等细分）是给**多人多线程**协作设计
- {{PROJECT_NAME}}是**单人多 PM 职责**工作流，项目 PM 统一验收 — 5+2 态足以覆盖
- 演化哲学第 1 条：不盲目对齐外部 framework

## 六、与议题状态的关系

| 任务状态 | 对应议题状态 |
|---|---|
| PLANNED | 议题登记（候选 backlog） |
| IN_PROGRESS | 议题实战中 |
| DELIVERED | 议题代码闭环 |
| ACCEPTED | 议题真机闭环 |
| CLOSED | 议题永久关闭（或归档监控）|
