# 交接卡格式附录

本附录承接低频解释；高频执行模板见 [`交接卡格式.md`](交接卡格式.md)。

## 为什么必须发 chat 简版

- PM / Owner 看 chat 时通常不会立即打开完整交接卡。
- 简版是视觉信号，让接手方秒判断“该 confirm 还是该跟进”。
- 长期运行后，①-⑦ 就是跨 PM 工作流的最低成本状态脉搏。

## 不要用总结段替代

即使内容详细，“总结段”（列文件 / 列测试 / 列决策的散文段）也不能替代 ①-⑦：

- 总结格式没有固定字段，PM 一眼看不到关键点。
- 必须用 ①-⑦ 编号，才能和完整文件基础 ①-⑥ + PM 切换轨迹对齐。
- 失败案例：F-PROFILE-12 曾给出长总结，但 PM 一眼看不出该确认什么、该跟进什么，触发 PROP-014。

## ⑦ PM 切换轨迹硬约束

触发历史：议题 AJ 留痕机制多次复发，反复证明“PM 自我观察”这种软规则会衰减。

硬约束规则：

1. 每次实施完成 / 跨角色 chat 简版输出必含 ⑦ 段。
2. 内容写“本次 session 加 N 行（状态.md L<line>）”。
3. 若 N=0 且本次 session 有 framework 改动，先补登 `状态.md` PM 切换轨迹再输出。
4. 纯简单对话本来不需要 chat 简版；如被 hook 误判收尾，可写“`N=0 / 本 session 无切帽子`”说明未发生角色切换。

目标是把软性的自我提醒转成模板硬触发，防止 PM 轨迹断档。

## 强制矩阵

| 行为 | 完整卡 | chat 简版 ①-⑦ |
|---|---|---|
| 跨角色切换 / 实施完成 | 强制 | 强制 |
| PM 内部推进 | `状态.md` 即可 | 可选 |
| 临时讨论 | 不需要 | 不需要 |

zlbdh 复制简版贴给另一端 session，接手者读 `交接区/待接手/` 完整文件后应能 1 分钟续上。

## 自动对齐为什么不静默改写

当前自动化故意收窄：

- `chat-summary-check`：只检查最终回复是否有 ①-⑦、⑥ 路径、⑦ 行号；失败时阻断或提醒，不改写回复。
- `handoff-zone-check`：检查待接手卡 `## ①`-`## ⑥` 标题顺序、⑤ `Status` 5 态、积压和状态断链；不会自动把卡从 `待接手/` 移到 `已接手/`。
- `pm-tracking-check`：检查 `状态.md` PM 轨迹新鲜度；不会替 PM 追加轨迹。
- `readme-index-check`：检查列表/计数一致性；只在确定性索引场景允许配套 Apply。
- ADR watcher：只自动维护 ADR README 这类确定性表格。

原因：交接移动、状态摘要、PM 轨迹和最终 chat ①-⑦ 都是语义证据，必须由项目 PM 主会话判断“真的完成 / 谁接谁 / 下一步给谁”，不能让 hook 猜。

## 向后兼容

读到旧 3 段格式（PROP-009 前 = 完成 / 下一步 / 警戒）可以继续接手，但新写卡至少基础 ①-⑥，待接手卡必须按 `## ①` 到 `## ⑥` 标题顺序落地。状态推断推断 5 在读到旧格式时提示“建议下次实施时升级到基础 ①-⑥”。

2026-06-14/15 早期已接手卡可能仍有旧式 frontmatter `status: HANDOFF`；不批量改历史事件流，由历史边界和到期归档覆盖。新写 / 新接收卡按 `pending` → `accepted` 执行。
