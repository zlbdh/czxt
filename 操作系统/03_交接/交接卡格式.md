# 工作流：交接卡格式

跨 session / 跨 PM 职责角色协作时，交接卡负责把“谁接手、改了什么、测到哪里、下一步做什么、风险在哪里”说清楚。Cowork / Claude Code / Codex 只是执行载体，文件名和正文主体优先写 PM 职责角色。

> 本文件是高频执行入口；历史触发、失败案例和强制矩阵详见 [`交接卡格式-附录.md`](交接卡格式-附录.md)。

## 自动对齐边界

hooks / 体检脚本只检查、提醒或阻断，不静默改交接卡、`状态.md`、PM 轨迹或最终 chat；只有 ADR README 等确定性索引可 Apply。交接移动、摘要和 ①-⑦ 必须由当前主会话按 PM 帽子确认后落笔。

## 三层结构

1. `交接区/待接手/YYYY-MM-DD-HHMM-任务-从到.md`：完整独立文件，基础 ①-⑥ 强制，可追加文件 ⑦。
2. `状态.md` 顶部：有待接手时链最新卡；无待接手时指向最新已接手或写“无待接手”。
3. chat 简版 ①-⑦：发给 zlbdh / 下一棒 PM 的收尾信号。

实际目录规则见 [`交接区/README.md`](../../交接区/README.md)。

## 完整卡模板

写入 `交接区/待接手/`，命名用 PM 角色；frontmatter `status` 必须先是 `pending`。

```markdown
---
status: pending
from: <发送方PM>
to: <接收方PM>
created: YYYY-MM-DD HH:MM
scope: framework/project
---
# <任务名> — <发送方 PM> 到 <接收方 PM>

## ① 时间 + 角色切换
- **时间**：YYYY-MM-DD HH:MM
- **从** → **到**：<发送方 PM 职责角色>（执行载体：<Cowork / Claude Code / Codex>）→ <接收方 PM / zlbdh>
- **任务**：本次实施的核心

## ② 文件变更清单
~~~
+ 路径（新建文件 + 字节）
M 路径（修改文件，旧字节 → 新字节）
- 路径（删除文件）
~~~
涉及 ≥6500B 必须说明；以 P4b / 人工复核为准。

## ③ 测试状态（端能力对照）
| 项 | 端 | 状态 | 结果 |
|---|---|---|---|
| vitest | Windows | ✅/❌/— | N/N |
| esbuild | Cowork/Win | ✅/❌/— | 0 fail |
| vite build | Windows | ✅/❌/— | 4.32s |
| dev APK | Windows | ✅/❌/— | path |
| 真机 smoke | Android | ✅/❌/— | 截图齐 |

## ④ 接手者待办（具体 checklist，不接受“测一下”）
- [ ] **zlbdh confirm** XXX
- [ ] 具体动作 1
- [ ] 具体动作 2

## ⑤ 警戒事项
Status: <DONE / BLOCKED / HANDOFF / RISK / OBSERVE>

- 🔴/🟡/✅ 描述（含大文件警报 / TODO / 风险）

## ⑥ 追加问答（接手者回炉问题时双向）
- 暂无

## ⑦ PM 切换轨迹 / 下一步（framework 卡建议；非 framework 可省略）
- 本轮需在 `状态.md` PM 切换轨迹追加/已追加 N 行。
```

完整文件硬底线是 ①-⑥；新写待接手卡必须使用 `## ①` 到 `## ⑥` 标题并保持顺序，⑤ `Status:` 只能取 `DONE / BLOCKED / HANDOFF / RISK / OBSERVE`。frontmatter `status`=流转态（待接手 `pending`，接收后 `accepted`）；正文 `Status:`=交付警戒态。framework 卡建议追加 ⑦；chat 简版强制 ①-⑦。

## 交付方完成实施

1. 把自己之前接的卡从 `交接区/待接手/` 移到 `交接区/已接手/`（表示“我接的事做完了”）。
2. 写新卡到 `交接区/待接手/YYYY-MM-DD-HHMM-任务-从我到下一个角色.md`。
3. 更新 `状态.md` 顶部摘要，链接到新卡。
4. 若切 PM / 改 framework，追加 `状态.md` PM 切换轨迹。
5. 在 chat 发简版 ①-⑦。

## 接手方起手

1. 读 `状态.md` 顶部摘要。
2. 读 `交接区/待接手/` 最新文件或用户指定卡。
3. 开始做事时不移动这张卡；完成时再按“交付方完成实施”流转。

## chat 简版 ①-⑦

实施完成时（写代码 / 改 framework / QA / push 等），除了写完整交接卡文件，还必须在 chat 里发紧凑版：

```text
【交接卡 - <任务名> 从 <A> 到 <B>】
① 时间：YYYY-MM-DD HH:MM
② 文件变更：N 新建 / M 修改（含关键 commit hash / 字节体检结果 / APK 路径）
③ 测试：vitest ✅/❌ N/N | esbuild ✅/❌ | build ✅/❌ | APK ✅/⏳ | smoke ✅/⏳ | push ✅/⏳
④ 你要做：[1] xxx [2] xxx [3] xxx（具体 checklist 不接受“测一下”）
⑤ 警戒：✅/🟡/🔴 关键警戒（含 ADR-016 6 条件合规情况）
⑥ 详情：读 交接区/待接手/YYYY-MM-DD-HHMM-<任务>-<从>到<到>.md
⑦ PM 切换轨迹：本次 session 加 N 行（状态.md L<line>）
```

## 强制约束

- 不能用“总结段”代替 ①-⑦；字段必须按 ①→⑦ 顺序出现。
- 有实施 / 跨角色触发 chat 简版时，⑥ 必须用纯文本路径指向 `交接区/待接手/` 下存在的 `.md` 文件；接收归档收尾且 `交接区/待接手/` 为空时，可指向 `交接区/已接手/` 下 `status: accepted` 的 `.md` 文件；不要写 Markdown 链接。
- 实施完成 / 跨角色触发 chat 简版时，⑦ 必须写 `状态.md L<line>`，且该行是 PM 切换轨迹表行；纯简单对话本来不需要交接卡，如被 hook 误判收尾，⑥ 可写“无”，⑦ 写 `N=0 / 本 session 无切帽子`。
- 旧 3 段格式可读；新写至少基础 ①-⑥，且待接手卡会被 `handoff-zone-check` 按标题顺序和 5 态 `Status` 自动检查。
