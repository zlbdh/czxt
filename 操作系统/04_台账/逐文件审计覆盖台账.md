# 逐文件审计覆盖台账

> 自动守卫通过不等于人工逐字覆盖。本文只记录“哪些文件/范围被什么方式检查过、证据在哪里、还缺什么”，不替代 `check-operating-system.ps1`。

## 一、记录口径

| 字段 | 说明 |
|---|---|
| 文件路径 / 范围 | 单文件、目录或明确 glob |
| 分类 | 活跃 / 历史 / 混合 |
| 检查方式 | 自动守卫 / explorer 只读 / worker 实修 / 主会话复核 |
| 覆盖粒度 | 全读 / 抽查 / 模式扫描 / 链接锚点 / 大小体检 |
| 证据 | agent id、状态行、交接卡、命令或守卫名 |
| 结论 | 通过 / 已修 / 误报 / 待复核 |

## 二、覆盖类型边界

| 类型 | 能证明 | 不能证明 |
|---|---|---|
| P4a-P4s 自动守卫 | 文件存在、计数、链接、锚点、部分语义模式、hooks 运行态、大小边界 | 全文语义都正确 |
| P4o 活跃 Markdown 扫描 | 活跃 Markdown 相对链接/锚点可解析 | 历史正文每句话都是现行口径 |
| 历史边界守卫 | 历史文件首屏有“不可复制执行 / 当前真源”提示 | 历史正文无需阅读 |
| explorer 全读 | 指定范围内人工语义问题发现 | 其他范围已覆盖 |
| 状态.md / CHANGELOG 摘要 | 本轮做了什么 | 逐文件证据本身 |

## 五、后续追加规则

- 每轮深审新增一行，能列到文件时列文件；范围太大时拆成子台账或注明抽查边界。
- 遇到“自动守卫通过”时，只能写对应守卫覆盖类型，不能扩大成全文人工语义证明。
- 如果后续真的完成某目录逐文件全读，必须记录 reviewer、时间、发现编号、处理结论和验证命令。
