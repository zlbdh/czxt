## 〇、顶层定位 — 这个操作系统是什么 / 不是什么

### ✅ 是

- **协作元-框架**：让 zlbdh + AI（Cowork PM / Claude Code / Codex）+ 6 PM 子角色高效协作的运行控制层
  - 🟢 **答 1：6 PM 是否够** — 当前小项目 ✅ 够用。业界 BMAD 12+ agents（Analyst / Architect / SM / UX / 创意 / 数据 / 安全等），我们缺 UX / 数据 / 安全 / 法务（小项目边际收益递减 / 暂不必加）
  - 🟢 **答 2：6 PM 是主 PM 吗** — ❌ 不是。**只有"项目 PM 咪咪"是主 PM（编排者 / 唯一对外面 / PM 自纠 #63 铁律）**。其他 5 PM = 项目 PM **内部切的执行帽子**，不是独立主 PM（我们应该拆分主-子-子子agent，我们应该合理划分PM）
  - 🟢 **答 3：业界最前沿范式**：① BMAD（12+ agents / Scale-adaptive）② Kiro（3 steering files + Hooks）③ Anthropic Skills（Progressive Context Loading SDK）④ Cursor（rules + AGENTS.md）— **本操作系统 5 项核心能力已对齐 / 但 SDK 集成层仍待工具升级（议题 CK / CS 待 backlog）**（规划升级方案并升级）
- **跨工具同步规范**：Cowork ↔ Codex ↔ Claude Code 三角协作机制 + 单一信息源原则
  - 🟢 **答 4：工具 vs PM 概念关系** — 部分对应但**不等同**：
    - ✅ **Cowork = 项目 PM 的载体**（Cowork 内部跑项目 PM 咪咪 + 切 5 帽子做 framework 治理）
    - 🟡 **Claude Code 当前 ≠ 开发 PM**（仅执行工具角色 / 不做决策） / **🟢 答 29 候选升级**：未来 v4.0 升级为「**开发 PM「Claude Code」**」— 因实战已有部分 PM 决策权（如 PROP-025 通过 decision-checkpoint 反向 review 挡下 PM Step 8 错向）。当前 5 PM 是 Cowork 内部帽子，**Claude Code 应独立为第 7 PM**（外部 PM 角色）

- **自我治愈系统**：通过 PM 自纠 + 议题 backlog + RETRO 不断演化（已累积 21 PM 自纠 / 63 议题 / 25 ADR）
  - 🟢 **答 5：自愈实现机制 4 层** —
    - **Layer 1（软规则）**：PM 切角色 → decision-checkpoint Q1-Q5 → 状态.md 留痕（⚠️ 9 次失效 / 实证靠不住）（软规则不一定会生效啊！）

  - 🟢 **答 6：需不需要"自愈 PM"** — ❌ 不需要独立 PM。自愈是**项目 PM 通过 6 帽子内部完成**（操作系统 PM 维护 framework + 测试 PM 验收 + 技术 PM 诊断）。专门的"自愈 PM"会违反 PM 自纠 #59 跨界禁令 + PM 自纠 #63 项目 PM 唯一对外身份铁律（我觉得我们需要这样一个“自愈PM”，你觉得呢？）
