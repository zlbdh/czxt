## 一、研究发现 — 业界标杆 4 家

### 1. BMAD Method（最高对标 / 46.7K stars / v6.6.0 / 2026-04-29）

**核心架构**：12+ 专门 AI agent / Agentic Planning + Context-Engineered Development 双核心 / 4 阶段渐进式 context build

**12 agent**：Analyst / PM / Architect / Developer / SM (Scrum Master) / UX / QA + game dev/创意创新等扩展

**关键创新**：
- **Scale-adaptive intelligence**：bug 修 = 轻量 workflow / 企业平台 = 全 ceremony（架构 review + sprint plan + 跨 agent 协作）
- **Verifiable artifact**：每 agent 强制产出可验证文档（PRD / 架构图 / 测试计划 / sprint story）— 不只是 chat response
- **Universal**：Claude / Cursor / Windsurf / Copilot 全兼容（MIT 开源）

### 2. Kiro（AWS / Amazon Bedrock）

**核心架构**：Spec-driven development（specs = unit of work）

**3 个 steering files**：
- `product` — 项目目的 + 关键能力
- `structure` — 组织原则 / 工具抽象模式 / 错误处理标准
- `tech` — 技术栈 + 开发指南

**关键创新**：**Hooks 事件驱动自动化** — file save / PR open / repo event 触发自动测试 / 更新文档 / 重新生成 fixture / 级联 spec 变更

### 3. Claude Code Skills 架构（Anthropic 官方）

**核心架构**：4 层文档金字塔 + Progressive Context Loading

| 文件 | 用途 | 加载时机 |
|---|---|---|
| `CLAUDE.md` / `AGENTS.md` | 项目持久 context | 每次 session 自动注入 |
| `MEMORY.md` | 决策日志 / 防 session amnesia | 总是 / on-demand |
| `SKILL.md` | 领域专长 playbook | **条件加载**（YAML front matter ~100 token 决定）|
| Context.md | 实时上下文片段 | 按需 |

**关键创新**：**Progressive Context Loading** — 30 skills × 100 token = 3K cost vs 全 load 30K（**10x 成本节省**）

### 4. Multi-Agent Orchestration 4 核心模式

| 模式 | 用途 | 业界采用率 |
|---|---|---|
| **Orchestrator-worker** | 主 agent 拆任务 + 多 worker 执行 + 聚合 | 最广泛 |
| Split-and-merge | 并行多分支 + 合并 | 高 |
| Planner-generator-evaluator | 规划 → 生成 → 评估闭环 | 中 |
| Consensus | 多 agent 投票 / 共识 | 低 |

**成本优化**：**模型分层** — cheap model triage + capable model 复杂推理 → 40-60% 成本节省

## 三、改进建议（按价值 × ROI 排序）

### 🔴 P0 — 改进 1：引入 SKILL.md 架构 + Progressive Context Loading

**痛点**：PM工作区/项目PM-咪咪/速查表/ 7 文件每次 PM 切角色都全 grep（always-load） / 当前 12.5KB / 未来扩到 30+ 速查表会爆炸

**升级方向**：
- 速查表/ → SKILL.md 模式（每文件加 YAML front matter）
- PM 切角色 Q1-Q3 中加 Q4「描述匹配的速查表」→ 条件加载
- 6 PM 子角色 × N 速查表 自然规模化

**收益**：context 节省 5-10x / 触发更精准 / 速查表数量可无限扩

**难度**：M / **建议 Sprint-8 PROP-030 起 PRD**

### 🔴 P0 — 改进 2：Hooks 事件驱动留痕（对症 PM 自纠 #62/#57/#58 反复触发）

**痛点**：议题 AJ 留痕机制 **9 次失效**（PM 自纠 #38/#41/#42/#54/#55/#56/#57/#58/#62）— "软规则 PM 自觉" 失效已实证 9 次

**升级方向（采纳 Kiro Hooks 模式）**：
- chat 简版输出后**自动 trigger**：扫 状态.md 末尾 PM 切换轨迹 / 距上次 >5 分钟 + 有 framework 改动 → 强制弹出"补登提示"
- 不依赖 PM 自我观察 → 机制硬约束

**收益**：解决 PM 自纠 #58「PM 自我观察衰减需外部 trigger」根因

**难度**：H（需 Cowork / Codex 支持 hook 接口 / 议题 CK 升级 PROP-027 v2）

### 🟡 P1 — 改进 3：PM 自纠独立 artifact 化

**痛点**：PM 自纠 #38-#63 共 26 次，全在状态.md / RETRO 散落 — 不可独立检索 / 不可独立 verify / RETRO 起稿时要重新汇总

**升级方向**：
- 建 `PM工作区/项目PM-咪咪/PM自纠/`（已在 README 规划但未实施）
- 每 PM 自纠一个 .md 文件（编号 + 触发场景 + 根因 + 防御机制 + 后续）
- 跨 Sprint 累积 + RETRO 自动汇总

**收益**：可独立检索 / verify / 跨 Sprint 同模式识别更快

**难度**：L / **可立刻执行**

### 🟡 P1 — 改进 4：跨 PM 共享技能库

**当前**：每 PM 私人速查表 + 跨界禁令 / 通用 SOP（如 git 撤销 / Cowork mount stale 恢复）在多个 PM 重复
**升级方向**：建 `操作系统/02_智能体/共享技能/` — 跨 PM 通用 SOP（与"PM 私人速查表"区分）
**难度**：M

### 🟢 P2 — 改进 5：Scale-adaptive 自动判定升级

**当前**：L1-L4 改动分级 由 PM 主观判定
**升级方向**：decision-checkpoint workflow 加 Q4「scale 判定」→ 自动选 workflow 路径（bug = skip PRD / Sprint 棒 = 标准 / 架构重构 = 全 ceremony）
**难度**：M

### 🟢 P2 — 改进 6：模型分层（节省成本 40-60%）

**当前**：业务代码 + framework 治理全 MiMo (mimo-v2.5-pro)
**升级方向**：业务代码（用户对话）保持 / framework 自动化 trigger 用轻量 model
**难度**：H / 候选议题 backlog
