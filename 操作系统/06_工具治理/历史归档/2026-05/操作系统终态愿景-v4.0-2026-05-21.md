## ⭐ 核心设计哲学（PM 自纠 #64 + #67-#72 累积 / 2026-05-22）

### 哲学 1：PM 角色 vs 工具载体彻底解耦（PM 自纠 #64）
- PM 角色 = 抽象协作角色
- 工具 = 可替换载体（Cowork / Cursor / Cline / Codex 等都是载体）

### 哲学 2：9 PM 4 大类 + 主-元-子-子子四层嵌套（PM 自纠 #68 + #72 / 2026-05-22 升级）

⭐ **PM 自纠 #72**：沉淀 PM **不是子子 PM** — 因为沉淀 PM 负责**整个项目的沉淀**（横跨项目全周期 + 跨 9 PM 协调）/ 升级为「**元-层 PM**」（紧贴主 PM / 独立元-决策层）

- **主 PM**（1 个 / 唯一对外身份）：项目 PM「咪咪」
- **⭐ 元-层 PM**（1 个 / 横跨全项目 / 紧贴主 PM）：🪞 沉淀 PM「沉淀者」
- **子 PM 决策层**（5 个 / 不动手 + 编排）：操作系统 / 产品 / 技术 / 测试 / 运营
- **子子 PM 实施层**（2 个 / 跨 PM 调度）：🔨 开发 PM「实施者」+ ✅ 测试发布 PM「闭环者」

### 哲学 3：三层沉淀架构（PM 自纠 #69 + #71）
- **Layer 1**：每 PM 私人沉淀（PM 自治）
- **Layer 2**：沉淀 PM 主沉淀（跨 9 PM 协调 + 元-层职责）
- **⭐ Layer 3**：项目沉淀（项目全周期 / 业务+产品+用户+市场）

### 哲学 4：双层验证架构（PM 自纠 #70）
- **Layer 1**：每 PM 私人验证维度（PM 自治验证自己 + 下游）
- **Layer 2**：测试发布 PM 主验证（跨 PM 协调 + 业务端到端）

### 哲学 5：项目全周期范围（zlbdh 截图标注）
- **不只是开发** — 覆盖立项 + 需求 + 设计 + 开发 + 测试 + 发布 + 运营 + 推广 + 数据反馈 + 演化

## 四、⭐ 元规则池详解（答 zlbdh L150）

### 什么是元规则池

**元规则池** = 项目的**永久化协作铁律集合** — 类比国家宪法 vs 普通法律：
- 普通规则 = 日常 PROP / Sprint 决策 / handoff 卡（可改可废）
- **元规则** = 永久现行的最高协作铁律（跨 Sprint 强制遵守 / 不轻易翻案）

### 升级路径

```
PM 自纠（实战教训）
     ↓ 累积 3+ 次同模式
候选元规则（议题字母表 A-CQ+）
     ↓ Sprint RETRO 评估
PROP 立稿
     ↓ zlbdh approve
ADR（永久档案）
     ↓ 写入元规则池
✅ 永久化元规则（强制遵守 / 跨 Sprint / 跨 PM / 跨工具）
```
