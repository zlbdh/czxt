# 实施循环 — 附录

> 主入口见 [`实施循环.md`](实施循环.md)。本文件承接低频细则，避免主入口过长。

## 设计阶段

### 数据

- 要新表 / 改 schema 吗？
- 要改 `useAppData` 加 actions 吗？
- 要 migration 吗？

### UI

- 入口在哪个 page、哪个位置？
- 用现有组件还是要新组件？
- 错误状态 / 加载状态 / 空状态是否齐全？

### 文件清单

列 `+ / M / D`，例如：

```text
+ src/shared/NewComponent.jsx
M src/features/health/Health.jsx
M src/shared/database.js
M src/shared/database.test.js
```

写到 `Docs/3-开发文档/sprint-N-技术拆解.md` 的对应需求段。

## 代码阶段

推荐顺序：

1. 数据层（database / defaults / shared helpers）
2. 组件层（components.jsx）
3. feature 层（features/*.jsx）
4. 测试

每改一个关键文件后做截断检查：

```bash
wc -lc src/path/file.jsx
tail -3 src/path/file.jsx
```

文件大小、mount 写入限制和拆分建议见 [`能力资产/rules/已知技术约束.md`](../../能力资产/rules/已知技术约束.md)。

## 测试阶段

三层全过：

1. esbuild 语法
2. vitest（旧测试不破，新测试通过）
3. vite build

不过就回去修，不能先报 done。

## APK 与 smoke

沙箱通常不出 APK。可走三条路：

- 测试发布 PM「闭环者」在可用载体上跑 `powershell -NoProfile -ExecutionPolicy Bypass -File .\build-apk.ps1`，归档到 `{{APP_REPO_DIR}}/apk/{{PROJECT_NAME}}-vX.Y.Z-debug.apk`
- zlbdh 在 Windows 本机手动跑同一脚本
- GitHub Actions 手动备用出包，详见 `Docs/5-运维文档/GitHubActions说明.md`

smoke test 按 `Docs/4-测试文档/手动测试用例.md` 过一遍；新功能必须特别测，有 bug 回修。

## 收尾细则

- `Docs/1-需求文档/需求历史.md` 移条目从 Sprint X → 已完成
- 顶层 `README.md` 有用户感知改动时同步
- 任务计划 / 交接状态标 completed
- 给 zlbdh 按 chat 简版 ①-⑦ 汇报：做了啥 / 测试结果 / APK 在哪 / 详情卡 / PM 轨迹
