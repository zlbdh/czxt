# 工作流：实施循环

需求确认后到代码上线的标准循环。本文只保留高频入口和硬约束；阶段细则、端能力完整表、APK/smoke 细节见 [`实施循环-附录.md`](实施循环-附录.md)，完整 DoD 见 [`实施循环-DoD.md`](实施循环-DoD.md)。

## ⭐ Session 起手必做

**每次新 session 起手**，先按 `AGENTS.md` + [`00_总入口`](../00_总入口.md) 完成状态 / 角色 / 交接 / 体检链路，再把 [`状态推断.md`](../../能力资产/skills/状态推断.md) 作为补充对账：

1. 读项目根 `状态.md`、总入口、角色边界和最新待接手卡。
2. 跑项目体检 / 必要的状态推断 — 看上次到哪了 + APK/进度对账。
3. 不打扰用户：把推断结果作为内部上下文，只在用户问起或发现冲突时主动报告。

session 中段（连续 30+ 交互后）建议再跑一次防衰减。

## ⭐ 流程优先三阶判定（PROP-014 / ADR-018）

任何实施前 / 实施中遇到流程问题（规则错位 / 警戒误报 / 文档不一致 / 路径错乱等）→ 按严重度分阶处理：

| 严重度 | 表现 | 行为 |
|---|---|---|
| 🔴 阻塞 | 影响代码 / 测试 / 实施落地 | 立刻停手修流程 → 开 PROP 治理 → 修完再继续业务 |
| 🟡 非阻塞 bug | 警戒误报 / 文档不一致 / 路径错，不修能继续但留隐患 | 业务继续 + 立刻进 RETRO backlog |
| 🟢 优化机会 | 换种方式可能更好，不修也无碍 | 业务继续 + 下次 RETRO 讨论 |

判断目标：不让流程绑架业务，也不让业务掩埋流程债。

## ⭐ 交接卡强制矩阵（PROP-014 / ADR-018）

实施收尾时，两张卡都必发：完整文件 + chat 简版 ①-⑦。

| 行为类型 | 必写完整交接卡？ | 必发 chat 简版 ①-⑦？ |
|---|---|---|
| ✅ 跨角色切换（PM 职责优先；Cowork / Claude Code / Codex 仅为执行载体）| 强制 | 强制 |
| ✅ 实施完成（写代码 / 改 framework / QA / push）| 强制 | 强制 |
| 🟡 PM 内部推进（写 PRD / 选方案 / 答问题）| 只更新 `状态.md` | 可选，>3 步任务建议发 |
| ❌ 临时讨论（confirm / 简单问答）| 不需要 | 不需要 |

完整格式见 [`交接卡格式.md`](../03_交接/交接卡格式.md)。chat 简版 ①-⑦ 也在该文件里。

## 阶段总览

```
需求 (PRD)
   ↓
设计 (数据 / UI / 文件清单)
   ↓
代码 (按文件改)
   ↓
测试 (vitest + esbuild + vite build)
   ↓
APK (测试发布 PM / Windows / GitHub Actions)
   ↓
smoke test
   ↓
CHANGELOG / 状态.md / 交接卡
   ↓
等下一个需求
```

| 阶段 | 必做产物 | 细则 |
|---|---|---|
| 设计 | 数据、UI、文件清单；写入 `Docs/3-开发文档/sprint-N-技术拆解.md` 对应需求段 | [`实施循环-附录.md`](实施循环-附录.md) §设计阶段 |
| 代码 | 数据层 → 组件层 → feature 层 → 测试；改文件后做截断检查 | [`实施循环-附录.md`](实施循环-附录.md) §代码阶段 |
| 测试 | esbuild 语法、vitest、vite build；失败回修，不报 done | [`实施循环-附录.md`](实施循环-附录.md) §测试阶段 |
| APK / smoke | 测试发布 PM「闭环者」承接（playbook：[`../02_智能体/测试发布PM-闭环者.md`](../02_智能体/测试发布PM-闭环者.md)），或 zlbdh Windows / GitHub Actions；真机 smoke 按手测用例 | [`实施循环-附录.md`](实施循环-附录.md) §APK 与 smoke |
| 收尾 | CHANGELOG、状态.md、交接区、chat ①-⑦；必要时 PRD/PROP 收档 | [`实施循环-DoD.md`](实施循环-DoD.md) |

## PM DoD 快速边界

- **操作系统 PM「框架管家」DoD**：文档 + 静态校验 + 项目体检 + 状态推断 + 交接卡 + 状态留痕。
- **开发 PM「实施者」DoD**：业务代码 + 测试代码 + vitest / 必要 build 自测报告；不 bump version、不出 APK、不真机 smoke、不默认 commit/push。
- **测试发布 PM「闭环者」DoD**：测试 + build + APK + 真机 smoke + 条件性 commit/push/tag；当前可由 Codex 承载，zlbdh 可手动复核。
- **工具能做 ≠ 当前 PM 可做**：权限以 [`角色边界.md`](../01_架构/角色边界.md) + [`git流程.md`](git流程.md) 为准。

**git commit/push 6 条件**（PROP-012 / ADR-016）：① 仅 main 分支 ② commit msg 真实 ③ 不 force / 不 rebase ④ 失败不重试 ⑤ 交接卡明示 hash ⑥ contextual 授权。

实施完成报告必须明示：我跑了哪些项 / 待 zlbdh 跑哪些项。

## Definition of Done

DoD 完整清单（L1-L4 + 跨 session 同步包 + 失败回退）已拆出到独立文件 [`实施循环-DoD.md`](实施循环-DoD.md)。

**核心 6 项（必勾）速查** — 详细规则见 [`实施循环-DoD.md`](实施循环-DoD.md)：

- [ ] **decision-checkpoint** Q1-Q7 跑过（PROP-020 路径 D / 议题 AJ 错向防御 + ADR-038 agent 实例化判定）— 详见 [`decision-checkpoint.md`](decision-checkpoint.md)
- [ ] **代码** 按 PRD 实现，无 TODO 残留
- [ ] **测试** vitest + esbuild + vite build 三层全过
- [ ] **PRD** 状态「实施中」→「已完成」+ 需求历史加条目
- [ ] **PROP 收档** 状态字段切「已完成」+ mv + README 计数更新
- [ ] **跑两个对账 skill** 项目体检 + 状态推断 全过

L1/L2 简化清单 / 扩展项 / 跨 session 同步包 / 失败回退 → 见 [`实施循环-DoD.md`](实施循环-DoD.md)。
