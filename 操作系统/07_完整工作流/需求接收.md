# 工作流：需求接收

zlbdh 发来一个新需求时，咪咪按这个流程跑。

## 第 1 步：分类

判断需求大小：

```
单个文案 / 颜色 / 单文件小改     → 小改：直接进开发 PM 实施循环
新功能 / 新模块 / 多文件 / 数据 schema 改  → 大改：先写 PROP 到 `确认改动/待审批/`
不清楚                            → 向 zlbdh 提 1-3 个澄清问题（按当前载体可用工具执行）
```

## 第 2 步：确认理解

复述需求给自己 1 句话。如果说不清，那就是没听懂。

例：
- zlbdh：「记账列表能不能按日历看」
- 复述：「想要在记账页加一个月历视图，点格子看那天的明细」 ✓

## 第 3 步：检查清晰度

5 个必问：
1. 这是哪个 feature？
2. 数据怎么改？
3. UI 入口在哪？
4. 边界情况怎么办（无数据 / 错误 / 加载）？
5. 算 P0 / P1 / P2？

任何「不知道」立刻向 zlbdh 澄清，不要臆测。

## 第 4 步：写需求条目

按规模落档：

- **大改 / L3-L4**：先写 `确认改动/待审批/PROP-XXX-...md`，等 zlbdh 审批；审批通过后按 [`审批与归档.md`](审批与归档.md) 进入 PRD / ADR。
- **小改 / 已审批需求**：再进 `Docs/1-需求文档/` 对应 Sprint 文件，用业务语言写用户故事 + 验收标准 + 工作量 + 影响文件。

原则：不在 `确认改动/` 生命周期里的大改，不直接进 `Docs/1-需求文档/`。

## 第 5 步：让 zlbdh 过目

如果是大改：把 PROP 草案贴出来或链接给他，等他明确审批。

如果是小改：说「这个我自己直接做了，做完汇报」。

## 第 6 步：加 task

用当前载体可用的计划/任务记录方式把分解写下来，每步对应一个 task。

## 第 7 步：交给开发 PM

切换到开发 PM「实施者」或对应执行载体，按 `操作系统/02_智能体/开发PM-实施者.md` 与 `能力资产/rules/写代码.md` 执行。

## 例子完整跑一遍

zlbdh：「**首页能不能加一个『一句话感想』，每天写一句，AI 周末汇总**」

咪咪：

1. **分类**：新功能 + 数据 schema 改 → **大改**
2. **复述**：每天首页能写一句话感想，存起来，AI 周末整合成回顾
3. **清晰度**：
   - feature：Home + 数据 + AI ✓
   - 数据：新表？merge 进 chatMessages？ ❓
   - UI：Home 哪个位置？ ❓
   - 边界：周末没填怎么办？ ❓
   - 优先级：你定 ❓
4. **澄清问题** 问 4 个：
   - 你想存多久？（永久 / 30 天）
   - 想放在 Home 哪里？（聊聊上面 / 习惯下面 / 单独 Card）
   - 周末没填的天 AI 怎么处理？（跳过 / 提醒）
   - 这个比 Sprint 1 的 8 个需求更优先吗？
5. **拿到回答**，先写 PROP 到 `确认改动/待审批/`
6. **任务分解** 列代码改动
7. **等 zlbdh 审批**
8. **审批通过后** 按 `审批与归档.md` 建 PRD / ADR，再交给开发 PM「实施者」干活
