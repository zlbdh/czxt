## 建议怎么做

三件事，**轻量优先、硬规则只一条、退役档位列为可选**：

1. **04_台账加一张轻量「演化体征」表**（数据层新增一个聚合视图，不改 schema）：每篇 RETRO 收口时更新 N 个数字，把散落在演化哲学/Sprint节奏/各 README 的规模指标并列成时间序列——建议起步 5 列：① 永久元规则数；② 候选池数 & 最老候选滞留天数；③ ADR 现行数 & 累计被替代未退役数；④ 本轮 RETRO 区间 framework 改动批次数；⑤ RETRO 积压（候选"够格未升"数）。把"在乎复杂度"从定性宣示变成**可在 RETRO 一眼看趋势**的仪表。

2. **收敛硬规则一条**（流程层，加进 RETRO 模板/演化哲学的"必答项"）：**当永久元规则净增连续 2 个 RETRO 为正且期间无任何退役时，强制本轮 RETRO 回答一道问句——"有无该退役 / 合并 / 下沉的规则？"** 回答可以是"无，理由 X"，但必须显式回答，不能跳过。这是"提醒不阻断"，对齐 check-retro-cadence 的非阻塞哲学。

3. **（可选 / 备选）ADR + 候选池加"退役 / superseded"档位**：给 ADR 状态机和候选池各加一个收敛档位，让机制**支持"减"而非只能加**（被替代 ADR 可移入退役分区、候选可标"已废止"）。这条改动面更大，建议作 Phase 2，本 PROP 先不含。

- 数据层面：新增 1 个台账视图（指针+时间线式，不复制内容，与 INDEX.md L11 现有口径一致）；RETRO 模板加 1 道必答问句。
- 跟现有交集：与 check-retro-cadence.ps1（节奏提醒）互补——它管"该不该复盘"，体征表管"复盘时看什么"；顺带可推动统一"刀/批次/commit"计量口径（但不在本 PROP 强制范围）。

## 影响 / 代价（粗估）

- 工作量：**S–M**（一张台账表 + 一道 RETRO 必答问句；可选退役档位若做则升 M–L）。
- 影响文件数：~3–5（04_台账新增 1 视图 + INDEX.md 挂链 + RETRO/README 模板加问句 +（可选）ADR README 状态机说明 + 候选池维护口径）。
- 风险：**中低**——核心风险见下方"拒绝理由"（体征表自身可能变成新维护负担，反讽地增加复杂度）。
- 是否改 schema：**否**（纯文档/流程层；台账是聚合视图非数据模型）。

## 备选方案

- **A（本提案，推荐）**：体征表（5 列）+ 收敛硬规则（1 条必答）+ 退役档位列可选 Phase 2。轻、可触发、低维护。
- **B（最小版）**：只加收敛硬规则那一道 RETRO 必答问句，不建表——成本最低，但缺趋势可视化，"连涨 2 RETRO"得靠人工数，容易再次失守（正是当前 CD/CN/CP 滞留的成因）。
- **C（完整版）**：A + 立即落地 ADR/候选退役档位 + 统一"刀/批次/commit"计量口径 + 复杂度阈值上限——证据最足但 scope 大、改动面广，违背"PROP 是看看值不值得继续不是终稿"，建议拆分推进。

## 拒绝的话理由会是什么

- **最可能**："体征表本身又是一个要每篇 RETRO 维护的负担——为了治复杂度反而增加了复杂度，自指地坐实了它要解决的问题。" 这是真实风险，**规避办法已内建**：① 列数硬上限 5 列、② 数字全部从已有真源（元规则池.md frontmatter / ADR README / RETRO 索引）直接抄录非新算、③ 不达"连涨 2 RETRO"就不触发那道问句，平时零额外动作——让它是"读数仪表"不是"新台账工程"。若 zlbdh 仍认为边际收益不抵维护成本，可退到方案 B（只留硬规则问句、不建表）。
- 或希望先把"刀/批次/commit"三套计量口径统一了再上仪表（→ 那应先做计量口径对齐的前置 PROP，本表暂用"批次"占位）。

## 不在范围

- 不引入阻塞性硬闸（永远提醒不阻断，对齐 check-retro-cadence exit 5）。
