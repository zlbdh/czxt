# 规则：写代码（命名 / 文件大小 / 状态管理 / 数据流 / 错误处理）

跟 开发 PM「实施者」配合用。这里是更具体的代码风格 / 模式 / 反模式；当前真文件就在 `能力资产/rules/写代码.md`，不要再找旧的 `能力资产/skills/写代码.md`。

⚠️ **文件大小规则（PROP-013 / ADR-017 后按工具分层）**：

| 工具 | 红线 | 触发后怎么办 |
|---|---|---|
| **Cowork**（mount 截断风险）| 6500B 软建议 | >6500B 改大文件**用 Python/Bash 写入**（绕 Edit）；项目体检告警但不强制拆 |
| **Claude Code / Codex**（本地）| **无硬限制** | 按工程审美自决 |
| **所有工具共同** | 8KB 软建议抽组件 | >8KB 建议拆分（单一职责工程审美，跟工具能力无关）|

**为什么按工具分层（历史背景）**：
- PROP-007 P1-3 / ADR-010 原本是「≥6500B 必拆门槛」，源于 Cowork mount 截断（9KB 硬截断）的安全裕度
- 但泛化到 Claude Code / Codex 后变成「人为限制工具能力」（他们没 mount 限制）
- RETRO-005 卡 2/3：bash `wc -c` mount 缓存反复制造"幻警戒" + 多次无意义抽组件
- PROP-013 / ADR-017 拆分到合理粒度

**适用范围（不变）**：
- ✅ 适用：`{{APP_REPO_DIR}}/src/` + `操作系统/` + `能力资产/`（Docs/ / 历史档案豁免）
- ❌ 不适用：`Docs/` + `确认改动/已审批/已完成/` + `交接区/已接手/` + `交接区/历史归档/`（低频写入 + 历史档案）

当前警戒清单见 [`能力资产/skills/项目体检.md`](../skills/项目体检.md) 维度 1。

## 命名

| 元素 | 风格 | 例子 |
|---|---|---|
| React 组件 | PascalCase | `TaskDetailModal` |
| React 文件 | PascalCase.jsx | `Health.jsx` |
| 工具函数 | camelCase | `parseLedgerText` |
| 工具文件 | camelCase.js | `ledger.js` |
| 测试 | xxx.test.js | `ledger.test.js` |
| 常量 | SCREAMING | `LLM_DEFAULTS` |
| 状态 hook | use- 前缀 | `useAppData` |

## 文件组织

- 共享 UI 组件按职责放在 `src/shared/`：基础结构在 `components.jsx`，按钮 / 输入 / 统计 / 月历 / 弹窗拆成独立小文件
- 共享工具 **按职能** 分文件（ai / health / ledger / date / image / devices）
- feature 主组件 `<FeatureName>.jsx` 负责组合、状态和 handlers；超过安全区的子 UI 拆到同目录小组件

## 状态管理

- 全局状态 ← `useAppData()` 唯一来源
- 不在组件里直接 `db.xxx`
- 局部 UI 状态 ← `useState`
- 派生数据 ← `useMemo`

## 数据流

```
用户操作 → actions.xxx() → db.xxx → setData → re-render
```

不要：
- 直接 `setData` 而不写 db
- 直接 `db.xxx` 而不 `setData`
- 在组件里用 `await db.xxx`

## 错误处理模式

```jsx
const [error, setError] = useState('');

async function submit() {
  setError('');
  try {
    await actions.addRecord(...);
  } catch (e) {
    console.warn('submit failed', e?.message?.slice(0, 120));
    setError('操作失败，请重试');
  }
}

// UI
{error && <div className="...">{error}</div>}
```

## AI 调用模式

```jsx
const [loading, setLoading] = useState(false);

async function fn() {
  setLoading(true);
  try {
    const result = await callAI({ ... });
    // 用 result
  } catch (e) {
    // 显示友好错误
    console.warn('ai call failed', e?.message?.slice(0, 120));
    setSomething('AI 暂时没接上，请稍后再试');
  } finally {
    setLoading(false);
  }
}
```

## React 最佳实践

- 默认导出页面组件 `export default function HomePage`
- 命名导出共享组件 `export function Card`
- 派生数据用 `useMemo`，不要重复计算
- ref 用 `useRef`，不要 `document.querySelector`
- 副作用用 `useEffect`，不要在渲染里调

## 反模式（不要做）

- ❌ 在组件里 `setInterval` 而不清理
- ❌ 异步操作不 try/catch
- ❌ 直接修改 props
- ❌ 用 index 当 key
- ❌ inline 样式写一大坨（超过 5 个属性 → 用 className）
- ❌ 一个文件长期承担多职责（优先按职责拆；200 行只是审美提醒，不是硬门禁）
- ❌ 在 useEffect 里写 setState 不加依赖

## 代码风格

- 缩进 2 空格
- 单引号字符串
- 末尾分号（保险）
- 函数名说"做什么" - `parseLedgerText` ✓ vs `process` ✗
- 注释说"为什么"，不说"做什么"（代码已经说了）
