## 跟 AI 边界三类（A/B/C）的对应

⭐ **L1-L4 管「怎么做」，A/B/C 管「该不该做」——两个体系正交但有重叠映射**：

| L 等级 | 边界类 | 含义 |
|---|---|---|
| L1 / L2 | **A 类（自动）** | 仅当 Q1-Q7、路径白名单、三类行为铁律均通过，且未命中 B/C 敏感清单时，才可按 A 类自动处理 |
| L3 / L4 | **B 类（必问）** | 新功能 / 架构变化，zlbdh 必须 review PRD（L4 还要 ADR）|
| 任意级别 + 常规 `git commit / push` | **B 类（条件性）** | 仅 `{{APP_REPO_DIR}}/` 主仓 main 分支，且必须满足 ADR-016 6 条件 |
| 任意级别 + `git push --force` / rebase 已 push / 删分支删 tag / 发分发渠道 release | **C 类（永不能动）** | 不可逆或发布对象决策，AI 不做 |
| 任意级别 + API key / 用户隐私 / 真机数据删除或外发 | **C 类（永不能动）** | 跟 L 等级无关，只要触碰就停手 |

→ A/B/C 当前真源见 [`操作系统/01_架构/三类行为铁律.md`](../../操作系统/01_架构/三类行为铁律.md)，L1-L4 对照解释见 [`三类行为铁律-附录.md`](../../操作系统/01_架构/三类行为铁律-附录.md)。

## 等级误判怎么办

如果开了 L2 跑到一半发现要改 schema，**立刻停下升级到 L4**，不要硬塞。
理由：L2→L4 跳级会导致 PRD 缺失、ADR 没写，将来没人能从文档追溯为什么这么改。

升级的步骤：
1. 把当前进度写到 PROP（`确认改动/待审批/`），描述发现了什么、为什么需要升级
2. 等 zlbdh 确认升级
3. 按 L4 流程从头补齐 PRD + ADR
