# 规则：改动分级（L1-L4）

把现有「小改 / 大改」二分扩展成四级，并把**测试问题分诊**也并进来。
每次开始一个改动前，咪咪先判断等级写在 task 描述里，让 zlbdh 一眼知道是否需要他过目。

---

## 四级速查表

| 等级 | 描述 | 例子 | 流程 | 谁拍板 |
|---|---|---|---|---|
| **L1** | 文案 / 单点小修 | 改副标题、修 typo、调一个颜色参数、删一个无用 console.log | Q1-Q7 + 路径白名单 + 三类行为铁律通过后，对应 PM 直接改 + 一行 CHANGELOG | 对应 PM 自决（未命中 B/C），项目 PM 验收 |
| **L2** | 行为微调 / 单点 bug | 调整已有功能的小行为、修一个具体 bug、加错误提示 | Q1-Q7 + 路径白名单 + 三类行为铁律通过后，改 `Docs/3-开发文档/` 相关章节 + 代码 + 测试 + CHANGELOG | 对应 PM 自决（未命中 B/C；提交/发布仍走 ADR-016） |
| **L3** | 新功能 / 新模块 | 任何 `F-XXX` 编号的新功能、跨多文件改动、UI 入口变化 | 走完整三件套：产品 PM 写 PRD → 开发 PM 实施 → 测试 PM / 测试发布 PM 验证 → 出 APK → smoke | **zlbdh 必须 review PRD** |
| **L4** | 架构 / 数据 schema 变化 | 加新表、改 schema 字段、加新 tab、拆模块、换 router、加原生插件 | L3 全套 + 写 ADR（`Docs/3-开发文档/adr/`） | **zlbdh 必须 review PRD + ADR** |

## 测试问题进来后的分诊

测试出 bug / 异常 / 不符合预期的现象 → 不要直接修，先**分诊**：

```
测试问题
  ↓
评估问题性质
  ├─ 单点修复（行为没变化，只是逻辑/边界 bug）   → L2
  ├─ 修复需要调整产品行为 / 用户感知会变化         → L3（走需求流程，重新写 PRD）
  └─ 修复涉及架构 / 数据 schema 改动             → L4（必须写 ADR）
```

**为什么要分诊**：
没有分诊的话很容易"以修 bug 之名加新功能"，最后 PRD 跟代码不一致。

---

## 跟现有命名的对应

| 现有用法 | 新等级 |
|---|---|
| 「小改」 | L1 + L2 |
| 「大改」 | L3 + L4 |

二分继续可用，L1-L4 是细化版。Sprint 是另一个维度（**时间/批次**），跟改动等级是正交的：一个 Sprint 里通常包含几个 L3 + 几个 L2。

---

## 用法 — 怎么打标签

每次新建 task 时把等级写在标题最前面方括号里：

```
[L1] 把咪咪的副标题改成"自我养成计划"
[L2] 修复 Health 月历 X 轴标签重叠
[L3] F-001 月历点击查看当天明细
[L4] F-XXX 加用户自定义任务模块（涉及新表 customTasks）
```

zlbdh 看到 L3/L4 标签就知道**要等他 review 才能往下走**；看到 L1/L2 知道对应 PM 可按白名单推进。
