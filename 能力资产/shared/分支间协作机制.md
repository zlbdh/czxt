# 分支间协作机制（双 PM 共享硬规则）

> 适用对象：**项目 PM 咪咪**（向内 / 业务-开发分支 / 9 PM 矩阵）+ **运营咪咪**（向外 / 商业-增长分支）+ 未来 PM 分支。
> 本文的“分支”指 PM 职责分支，不是 git branch；代码仓库仍按单 `main` 策略运行。
> 触发条件：跨分支信息流转、决策冲突、共享资源、人设一致性、用户反馈反哺。
> 关联：[PROP-011 交接区](../../确认改动/已审批/已完成/PROP-011-2026-05-11-交接区机制.md) + [ADR-023 PM 子类化](../../Docs/3-开发文档/adr/ADR-023-议题AJ落地-PM角色子类化+decision-checkpoint.md)。

## 一、分支结构

```
zlbdh（决策者 + 唯一仲裁人）
  │
  ├─ 向内分支 — 项目 PM 咪咪
  │     ├─ 操作系统 / 产品 / 技术 / 测试 PM
  │     ├─ 沉淀 PM
  │     └─ 开发 PM / 测试发布 PM（实现与发布执行角色）
  │
  └─ 向外分支 — 运营咪咪
        （未来：客服 PM / 商业化 PM / 社区 PM）
```

注：运营 PM「运营咪咪」是外向 GTM 例外，不在向内分支重复归属。

## 三、跨分支交接区（类比 PROP-011 事件流）

```
交接区/分支间/                       ← 新增（与「待接手/」「已接手/」平级）
  ├─ 项目PM→运营咪咪/
  │   ├─ 待处理/
  │   └─ 已处理/
  └─ 运营咪咪→项目PM/
      ├─ 待处理/
      └─ 已处理/
```

## 四、典型场景流程

| # | 场景 | 触发方 | 路径 | 决策方 |
|---|---|---|---|---|
| 1 | ship / ADR / RETRO 可做内容 | 项目 PM | 登记到项目→运营待处理卡 | 运营咪咪定平台 / 深度 |
| 2 | 用户反馈触发产品改进 | 运营咪咪 | 发起运营→项目待处理卡 | 项目 PM → 产品 PM 拆 Sprint |
| 3 | 内容涉及 App 截图/代码/隐私 | 运营咪咪 | 发起内容 review 卡 | **zlbdh approve**（保密一票否决）|
| 4 | 人设不一致争议 | 任何一方 | 提修改建议；操作系统 PM 落笔 shared | zlbdh approve |
| 5 | 决策冲突 / 优先级争议 | 任何一方 | 各自 chat 给 zlbdh | **zlbdh 仲裁** |

## 七、冲突仲裁原则

- **保密 vs 内容曝光争议** → 运营咪咪 yield，zlbdh 一票否决保密
- **节奏 vs 内容时机争议** → 项目 PM 决定 ship 时机，运营咪咪决定内容上线时机（互不干预）
- **人设偏离争议** → `咪咪人设统一基准.md` 是真相，谁偏离谁修正
- **资源 vs 投入争议** → zlbdh 仲裁
- **路径越界争议** → 越界方立刻 stop + 修 playbook（议题 AJ 教训）

## 八、收档与扩展

3-5 周 B 阶段实战 + 0 越界 + 跨分支交接 ≥ 5 次 0 冲突 → RETRO 评估是否新建 ADR（编号以 ADR README 为准；不得复用 ADR-024）。

未来加客服 PM / 商业化 PM / 周对齐机制时沿用同一事件流；若有结构性缺漏，另起 PROP 修。

## 九、不要做（硬反例）

- ❌ 碰对方专属目录（运营咪咪不碰 `{{APP_REPO_DIR}}/`、`Docs/3+7`；项目 PM 不碰运营工作区）
- ❌ 跨分支信息只走 chat 不落交接区
- ❌ 绕过分支间交接区直接向对方私区 drop 内容
- ❌ 直接修对方 playbook（只能提建议给 zlbdh approve）
- ❌ 私建或分叉项目记忆真源
- ❌ 自行裁决冲突（zlbdh 仲裁权不可代行）
