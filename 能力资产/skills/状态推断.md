# 技能：状态推断

让 AI **主动**对账文档状态 vs 实际现状，找出"应该切换但还没切的状态字段"。

## 何时跑

- ⭐ **实施循环新 session 起手时必跑**；先完成 AGENTS 5 步，本 skill 不能替代起手链路
- ⭐ **PROP 收档前必跑**（DoD 强制项）
- zlbdh 主动「@咪咪 跑下状态推断」
- 长 session 中段（每 30+ 交互）防衰减

## 跟项目体检的区别

| | 项目体检 | 状态推断 |
|---|---|---|
| 关注 | 框架自洽（标题 / 计数 / 死链 / 大文件） | **进度对账**（代码 vs 文档） |
| 输出 | 3 色报告（✅/🟡/🔴） | **建议清单**（"应该切 ✅"/"应该写 RETRO"） |
| 频率 | PROP 收档前 / 主动 | session 起手 / PROP 收档 / 发布闭环 / 用户主动 |

两个 skill 互补：体检看文档健康，推断看文档是否反映真实进度。

## 推断 10：状态.md 30 天新鲜度（PROP-017 加）

### 问题

AI 跨 session 起手会读 `状态.md` 作为现状基线；若 `状态.md` stale（>30 天没动），会导致决策偏差。

### 检测逻辑（多运行时口径）

权威入口：`能力资产/tools/scripts/check-operating-system.ps1` 的 P4d。Bash/GNU 跑法仅作 fallback，完整旧示例见 [`状态推断-跨session监控-附录.md`](状态推断-跨session监控-附录.md)。

```bash
# fallback：先切到当前项目根
cd "<项目根>"

now=$(date +%s)
threshold=$(date -d '30 days ago' +%s)
mtime=$(stat -c %Y 状态.md 2>/dev/null)
[ -z "$mtime" ] && echo "🔴 状态.md 不存在" && exit
days=$(( (now - mtime) / 86400 ))
[ "$mtime" -lt "$threshold" ] && echo "🟡 mtime stale ($days 天)" || echo "✅ mtime 距今 $days 天"

latest=$(grep -oE '[0-9]{4}-[0-9]{2}-[0-9]{2}' 状态.md | sort -ru | head -1)
echo "内容最新日期: $latest"
```

### 输出格式（AI 起手时）

```
🔍 推断 10 — 状态.md 新鲜度
  mtime: 2026-05-12（距今 0 天）✅
  内容最新日期戳: 2026-05-12（距今 0 天）✅
  → 状态.md 新鲜，可信
```

如 stale：

```
🟡 推断 10 — 状态.md 已 stale 41 天
  → 建议先更新 状态.md 再继续；否则我可能基于陈旧信息建议
```

### 与 PS1 P4d 的关系

- **PS1 P4d** = Windows host 真值，权威实现。
- **推断 10** = AI 起手对账；优先采用 PS1/host 真值，Bash/GNU 仅 fallback。
- 两者必须保持 30 天阈值 + mtime/内容日期双维度一致。

---

## ⚠️ 已知限制

- **mount 缓存延迟**：Bash 看到的 mtime / 文件大小可能滞后；验证信 Windows host 真值。
- **AI 是响应式**：状态推断只在「AI 被触发响应」时跑，不是真的 daemon 监听。
- **建议而非断言**：推断结果是「建议你做 X」，**最终决策权仍在 zlbdh**。AI 不因推断自动改状态字段；仅在用户确认或 PM 轨迹规则要求时写 `状态.md`。

## 🚫 防重复推荐机制（PROP-006 加）

跑出建议后必查 `状态.md`「最近建议 + 用户决议」段：
- 如果同样建议在过去 7 天内已 dismiss → **不重推**
- 如果状态是 ⏳ pending → 提一次但不啰嗦
- 用户明确给出决议且需要留痕时，再按 `状态.md` 规则追加；普通回应不自动写状态字段。
