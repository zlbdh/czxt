# 项目体检 — 检查项 5-6 细则

> 主文件见 [`项目体检.md`](项目体检.md)。
> 检查项 1-4 见 [`项目体检-检查项.md`](项目体检-检查项.md)。

## 检查项 5 / 9：跨 session 同步状态

### 病灶

Cowork、Claude Code、Codex 是独立 session，没有自动通道。完工时容易只改代码或卡片，不同步 PRD / 历史 / ADR。

### 检查规则

- 当前各 Sprint 需求清单里所有 ✅ 已完成的 F-XXX，需求历史.md 应有对应条目。
- 需求历史.md 里所有 PROP-XXX 收档记录，对应 PROP 文件应在 `已审批/已完成/`。
- ADR README 索引表里所有「现行」ADR，对应文件头部状态字段也是「现行」。

### 命令

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File 能力资产/tools/scripts/check-operating-system.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File 能力资产/tools/scripts/check-readme-indexes.ps1
```

交接区健康脚本只覆盖检查项 6，不覆盖 Sprint 清单 vs 需求历史 / PROP 收档 / ADR 状态一致性。

Bash/GNU fallback：

```bash
cd "<项目根>"

echo "--- Sprint ✅ F-XXX 集合（所有 Sprint 清单）---"
grep -Eh "^\| F-[A-Z0-9-]+" Docs/1-需求文档/Sprint-*需求清单.md | grep "✅"

echo "--- 需求历史 F-XXX 已完成集合 ---"
grep -E "✅ \*\*F-[A-Z0-9-]+" Docs/1-需求文档/需求历史.md

echo "--- 起草中的 ADR（应=0，除非真在写）---"
grep -l "状态.*起草中" Docs/3-开发文档/adr/ADR-*.md
```

### 例外

- 「待 APK + 真机 smoke」是合法等待状态，但 Sprint 顶部表格状态列要标 ⏳，不能标 ✅。

## 检查项 6 / 9：交接区健康

### 检查规则

- `交接区/待接手/` 卡数 ≤ 3。
- `交接区/待接手/` 新写卡应至少包含 `## ①`-`## ⑥` 基础标题，顺序不能乱；⑤ 内 `Status:` 只能是 `DONE / BLOCKED / HANDOFF / RISK / OBSERVE`。
- `交接区/已接手/` 超 30 天的卡建议归档；自动检查优先按文件名日期判断，避免 mtime 被触碰后误绿。
- `状态.md` 顶部摘要应和 `交接区/待接手/` 最新卡一致；如果待接手为空，摘要应说明当前无待接手。

### 命令

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File 能力资产/tools/scripts/check-handoff-zone.ps1
```

Bash/GNU fallback：

```bash
cd "<项目根>"

handoff_count=$(ls 交接区/待接手/*.md 2>/dev/null | wc -l)
echo "待接手卡数: $handoff_count"
[ "$handoff_count" -gt 3 ] && echo "  🟡 积压（≤3 才安全）"

threshold=$(date -d '30 days ago' +%s)
for f in 交接区/已接手/*.md; do
  [ -f "$f" ] || continue
  mtime=$(stat -c %Y "$f")
  [ "$mtime" -lt "$threshold" ] && echo "  🟡 $f 超 30 天，建议归档"
done
```

### 判定

- ✅ 待接手卡数 ≤ 3、待接手卡 `## ①`-`## ⑥` 顺序齐且 Status 合法，并且无超期卡。
- 🟡 待接手卡数 > 3、待接手卡缺基础标题 / 顺序乱 / Status 非 5 态，或存在超 30 天已接手卡。
- 🔴 状态摘要指向不存在的卡，或待接手卡与状态摘要互相矛盾。
