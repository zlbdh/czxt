## 五、依赖升级策略

### 何时升级
- ⚠️ **不主动升级**（议题 AT / 议题 BR confirm 模式 — 锁定版本防意外破坏）
- ✅ **C 类决策**：升级主版本需 zlbdh approve（同 ADR-016 force push / tag / API key 同级别）
- ✅ **B 类决策**：升级 patch / minor 可走 PROP 流程（如 PROP-022 加 @capacitor/network）

### lockfile

**`{{APP_REPO_DIR}}/package-lock.json`**（git tracked）— 锁定确切版本，跨设备一致。

## 六、添加新依赖流程

新依赖添加流程（议题 AT 防御）：
1. **产品 PM 起 PRD** 明示需要的功能
2. **技术 PM 评估**：是否有替代 / 版本约束 / 主版本兼容
3. **zlbdh approve B 类**（新增依赖默认 B 类）
4. **PM 起 PROP** 记录决策（如 PROP-022 @capacitor/network）
5. **开发 PM「实施者」实施** + 单测验证
6. **测试发布 PM「闭环者」ship**（含 lockfile 同步）

历史新增依赖：
- @capacitor/local-notifications（F-ALARM-1）
- @capacitor/network（F-SYSCHECK-1 / PROP-022）

→ 议题 AT 元规则详见 [web-api-信源选型.md](../rules/web-api-信源选型.md)
